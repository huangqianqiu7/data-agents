## 2. 数据格式规范

### 2.1 完整目录结构

评测期间,容器内部的完整目录结构如下。参赛者的 Agent 必须遍历 `/input` 下的所有任务子目录,逐个处理,并将结果写入对应的 `/output/task_<id>/` 目录。

```
/input/                          # 只读挂载,不允许写入
└── task_<id>/                   # 每个任务一个子目录,id 是唯一标识符
    ├── task.json                # 任务元数据(task_id、difficulty、question)
    └── context/                 # 任务所需的上下文数据
        ├── csv/                 # 可选:结构化 CSV 文件
        ├── db/                  # 可选:SQLite 数据库文件
        ├── json/                # 可选:结构化 JSON 文件
        ├── doc/                 # 可选:数据文档(Markdown 等)
        └── knowledge.md         # 可选:背景知识文档

/output/                         # 可读写,参赛者写入预测结果
└── task_<id>/                   # 与输入任务目录一一对应
    └── prediction.csv           # [必需输出] 参赛者的预测答案
```

**注意:**

- `context/` 下的具体子目录对每个任务并非固定;不同任务可能包含不同的数据源组合。每个任务至少包含 `csv/`、`db/`、`json/`、`doc/` 中的一项,可能还包含 `knowledge.md`。
- `/input` 是只读挂载;任何写操作都会导致运行时错误。`/output` 目录由评测系统预先创建,参赛者只需写入 `task_<id>/prediction.csv`,无需创建父目录。

### 2.2 task.json 格式

每个任务的任务元数据存储在 `task.json` 中,字段说明如下:

| 字段名       | 类型   | 说明                                                       |
| ------------ | ------ | ---------------------------------------------------------- |
| `task_id`    | string | 任务唯一标识,与 `/input/task_<id>/` 目录名一致             |
| `difficulty` | string | 难度级别:`easy` / `medium` / `hard` / `extreme`           |
| `question`   | string | 自然语言分析问题,描述需要回答的数据分析任务               |

`task.json` 示例:

```json
{
  "task_id": "task_11",
  "difficulty": "medium",
  "question": "Which product category generated the highest total revenue in Q3? List each category and its total revenue, sorted by revenue descending."
}
```

### 2.3 context/ 数据源说明

`context/` 目录下的数据源类型因任务而异。参赛者的 Agent 必须**检测并读取实际存在的**子目录和文件:

| 子目录          | 内容类型       | 说明                                                                |
| --------------- | -------------- | ------------------------------------------------------------------- |
| `csv/`          | 结构化表       | 一个或多个 CSV 文件,可直接用 pandas 等工具读取                     |
| `db/`           | SQLite 数据库  | 一个或多个 `.sqlite` / `.db` 文件,包含多张关系型表                 |
| `json/`         | 结构化数据     | JSON 格式的半结构化数据文件                                         |
| `doc/`          | 数据文档       | Markdown 或其他格式的数据报告与分析文档                             |
| `knowledge.md`  | 背景知识       | 与任务相关的背景知识文档,包含业务定义和术语解释                    |

**难度与数据源的对应关系:**

| 难度    | 数据模态                          | 文档规模               | 核心挑战                            |
| ------- | --------------------------------- | ---------------------- | ----------------------------------- |
| Easy    | CSV/JSON + `knowledge.md`         | 短文本                 | Python 代码生成与数据分析           |
| Medium  | CSV/JSON + `db/` + `knowledge.md` | 中等                   | Text-to-SQL + 多源数据分析          |
| Hard    | 同上 + `doc/` 数据文档            | 约 10K–128K tokens     | 非结构化文档推理                    |
| Extreme | 同 Hard                           | >128K tokens           | 长上下文工程与内存管理              |

### 2.4 输出格式:prediction.csv

对每个任务,参赛者的 Agent 必须将预测结果写入 `/output/task_<id>/prediction.csv`。该文件为标准 CSV 格式(UTF-8 编码),要求如下:

- 第一行为列名(表头);**列名不参与评分**,仅用于可读性。
- 第二行起为数据行,每行对应一条结果记录。
- **列顺序不影响评分**;评分基于列值向量的无序匹配(见第 6 节)。
- 数值数据应保留足够的精度;评测系统在比较前会将数值标准化为 **2 位小数**(四舍五入)。
- 空值应写为空字符串;参赛者写 `NULL`、`null`、`NaN` 等也会被评测系统统一标准化为空字符串。
- 对于姓名字段,**将姓和名拆为两列**或**合并为一列全名**都可接受;评测系统认为两种形式都正确。

`prediction.csv` 示例(对应问题:列出按品类的总销售额):

```csv
category,total_revenue
Electronics,4200000.00
Clothing,1850000.00
Food,930000.00
```

## 3. 提交方式

本比赛使用 **Docker 镜像**作为唯一的方案提交形式。参赛者需要将完整的 Agent 程序及其运行依赖打包成 Docker 镜像,导出为压缩归档,通过电子邮件提交给组织方。组织方收到镜像后,评测系统会在受控环境下统一启动容器、注入运行时配置、执行 Agent 并采集输出结果——全程自动化,无需参赛者干预。

### 3.0 提交前清单

组织方建议在每次提交前完成以下检查。许多第一轮失败都是由命名、平台、打包、共享权限、输出路径或 CSV 格式问题导致的。

- **镜像与归档名**:Docker 镜像 `<team_id>:v<N>`;归档 `<team_id>_v<N>.tar.gz`。必须符合 §3.1。
- **用 `docker save` 导出**:用 `docker save` 构建 `.tar.gz`。**不要用 `docker export`**(容器导出不是我们流水线可加载的镜像归档)。
- **linux/amd64**:评测在 linux/amd64 上运行。请确保镜像能在该平台启动(例如多架构构建,或构建时使用 `--platform=linux/amd64`)。
- **Google Drive**:共享必须允许组织方下载文件(如 "Anyone with the link" → Viewer)。你可以用自己的链接通过 `gdown` 验证下载。
- **输出路径**:在 `/output` 下,只有一层 `task_<id>` 目录,每个目录包含 `prediction.csv` —— 与示例布局对齐(§2.4)。**不要在 `/output` 内嵌套额外目录。**
- **有效的 CSV**:确认每个 `prediction.csv` 都是标准 CSV(UTF-8)且能被正确解析,以便评测器读取结果。
- **提交邮箱**:提交邮件必须从注册时登记的**队长邮箱地址**发出。其他发件地址不会通过验证,也不会被评测。

### 3.1 镜像命名规范

为确保提交镜像的唯一可追溯性,参赛者必须严格按以下格式命名 Docker 镜像与压缩归档:

| 项目          | 格式与说明                                                       |
| ------------- | ---------------------------------------------------------------- |
| **镜像名称**  | `<team_id>:v<N>`                                                 |
| **归档文件名**| `<team_id>_v<N>.tar.gz`(冒号在文件名中替换为下划线)             |

| `<N>`         | 提交序号,从 1 开始递增(如第 1 次提交为 `v1`,第 3 次为 `v3`)   |

**示例**(队伍 ID 为 `team0042`,第 3 次提交):

- 镜像名:`team0042:v3`
- 归档文件名:`team0042_v3.tar.gz`

> **注意**:镜像 tag 与归档文件名必须完全符合上述规范,`team_id` 必须与组织方分配的队伍 ID 完全一致。命名不合规的提交将被评测系统拒收。每次提交都必须递增版本号 `<N>`;此前已提交过的版本号不可重用。

### 3.4 评测系统执行命令

镜像上传后,评测系统会依次执行以下步骤:

**第 1 步:解压并加载镜像**

```bash
docker load -i team0042_v3.tar.gz
```

**第 2 步:启动容器进行评测**(概念示意,实际参数由评测平台生成)

```bash
docker run --rm \
  --network=eval_net \                              # 内部网络(只能访问评测系统服务)
  --cpus=16 \                                       # CPU 核心限制
  --memory=64g \                                    # 内存硬上限
  --memory-swap=64g \                               # 禁用交换空间
  -v /eval/data/input:/input:ro \                   # 所有任务输入(只读)
  -v /eval/<submission_id>/output:/output:rw \      # 所有任务输出(可读写)
  -v /eval/<submission_id>/logs:/logs:rw \          # 方案运行时日志文件(可读写)
  -e MODEL_API_URL=<model_url> \                    # 内部 LLM 模型服务地址
  -e MODEL_API_KEY=<api_key> \                      # 内部 LLM 模型 API Key
  -e MODEL_NAME=<model_name> \                      # 内部 LLM 模型访问名称
  team0042:v3                                       # 参赛者方案镜像名
```

> **注意**:`/input` 与 `/output` 都挂载整个数据集目录(包含所有任务)。容器必须遍历 `/input` 下的所有 `task_<id>` 子目录并逐个处理。

### 3.7 运行时日志规范

评测系统通过 Docker 挂载为每次提交分配独立的 `/logs` 目录(可读写)。当参赛者方案异常退出时,组织方可以提取 `/logs` 目录的内容供参赛者调试参考。

## 4. 硬件资源限制

### 4.1 资源配额

| 资源类型     | 限制规格                                                                       |
| ------------ | ------------------------------------------------------------------------------ |
| **CPU**      | 16 核(vCPU),x86-64 架构                                                     |
| **内存**     | 64 GB RAM;超出将被 OOM 杀掉,该次提交所有结果作废                              |
| **GPU**      | 参赛者容器**没有 GPU**;模型推理由评测系统统一完成(见第 5 节)                |
| **运行时限制** | 所有任务的总运行时长上限为 **12 小时**;超时后容器被强制终止                    |

> **注意**:12 小时是**所有任务的总时长上限**,而非每个任务的限制。请合理分配每个任务的处理时间,并在代码中实现超时保护,确保已完成任务的结果能及时写出。此外,我们建议参赛者使用多线程/多进程对不同输入样本进行批量并行加速(参考 Starter Kit 代码中的 `max_workers` 设置)。

### 4.2 超时与 OOM 处理

- **超时(12 小时)**:评测系统先发送 `SIGTERM`,等待 30 秒后若仍未退出则发送 `SIGKILL`。已写入的 `prediction.csv` 仍参与评分;未写出的任务记 0 分。
- **OOM Kill**:容器被强制终止;已写入的 `prediction.csv` 参与评分;未写出的任务记 0 分。
- **非零退出码**:**不影响评分**;评测系统只关心 `/output` 目录中是否存在文件。

## 5. 模型调用规范

### 5.1 开发阶段 vs 评测阶段

本比赛对模型使用采取 **"自由开发,统一评测"** 的策略:

| 阶段                    | 模型来源                              | 说明                                                                                            |
| ----------------------- | ------------------------------------- | ----------------------------------------------------------------------------------------------- |
| **开发 / 本地调试**     | 参赛者自选(任意 LLM)                | 组织方不提供开发 API;参赛者自行申请并使用各种模型服务                                          |
| **官方评测(提交后)**  | Qwen3.5-35B-A3B(由组织方统一部署)    | 评测系统在启动容器时通过环境变量注入模型服务地址;所有参赛者使用同一模型以确保公平              |

**关键设计原则**:参赛者代码必须从环境变量读取模型服务地址与 API Key,**不能将其硬编码在镜像中**。这样,同一份代码在本地开发时指向参赛者自己的模型,提交后会自动切换到组织方统一部署的 Qwen3.5-35B-A3B。

**公平性约束**:在官方评测期间,Qwen3.5-35B-A3B 是**唯一允许的主要任务求解 LLM**。参赛者不得在容器内运行任何其他 LLM 作为主要推理或生成引擎,**包括用 CPU 在容器内本地推理其他 LLM**。允许在官方硬件限制内使用合适规模的辅助模型用于文档处理、检索或表征(例如 embedding 模型),只要它们不被用作主要求解器。

### 5.2 运行时环境变量注入

评测系统在启动参赛者容器时,通过 `docker run -e KEY=VALUE` 的形式注入运行时环境变量。参赛者程序必须**在运行时**从环境变量读取这些配置,**不得在代码或镜像中硬编码** API URL、API Key 等信息。

| 环境变量名      | 说明                                                                                       |
| --------------- | ------------------------------------------------------------------------------------------ |
| `MODEL_API_URL` | 内部部署的 Qwen3.5-35B-A3B 模型服务地址(兼容 OpenAI Chat Completions API)                |
| `MODEL_API_KEY` | 模型服务认证 Key(由评测系统注入;参赛者无需事先知道具体值)                              |
| `MODEL_NAME`    | 评测时使用的模型名称,值为 `"qwen3.5-35b-a3b"`                                             |

> ⚠️ **警告**:**严禁在代码、配置文件或 Docker 镜像中硬编码 API Key、API URL 等敏感配置**。评测系统启动时注入的环境变量是唯一合法的配置来源。

### 5.3 组织方 vLLM 部署参考

为帮助参赛者在本地测试时与官方评测环境保持一致,组织方部署的 Qwen3.5-35B-A3B 服务在概念上以与下列命令等价的参数通过 vLLM 部署。其中涉及基础设施的敏感字段已用占位符代替。在评测期间,参赛者仍必须使用 `MODEL_API_URL`、`MODEL_API_KEY` 与 `MODEL_NAME`,而不是硬编码端点细节。

```bash
vllm serve <model_path> \
  --host <host> \
  --port <port> \
  --tensor-parallel-size 8 \
  --seed 1024 \
  --served-model-name qwen3.5-35b-a3b \
  --max-model-len 262144 \
  --reasoning-parser qwen3 \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_coder \
  --trust-remote-code
```

| 参数                                | 参赛者解读                                                                                                                                                                |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `<model_path>`、`--host`、`--port`  | 内部部署细节有意隐藏。请勿硬编码;始终从运行时环境变量读取实际端点。                                                                                                       |
| `--tensor-parallel-size 8`          | 官方后端使用 8 路张量并行提供模型服务。这是基础设施细节,不会改变参赛者代码看到的 API 协定。                                                                              |
| `--seed 1024`                       | 服务使用固定的后端 seed 以增强可复现性。参赛者仍应将输出视为依赖请求的,**不要假设跨不同提示或工具轨迹时输出完全确定**。                                                  |

## 6. 评分机制

### 6.1 评测流程说明

评测系统使用统一的自动化评测程序对参赛者提交的预测结果进行**离线批量打分**。参赛者需要按指定格式(如 `prediction.csv`)提交结果文件,系统会将其与对应的标准答案进行比较打分,生成最终得分。

所有提交均在同一评测环境中运行,评测过程为**确定性执行**,以确保评测结果的一致性与公平性。

### 6.2 列匹配方式

评测采用**列级内容一致性匹配方式**(column-level matching),核心流程如下:

- 读取 `prediction.csv` 与 `gold.csv`,解析列与数据内容
- 对每一列,排序该列的所有单元格值,构造"列签名"
- 分别统计列签名在 prediction 与 gold 中出现的次数
- 基于列签名进行匹配,计算 prediction 对 gold 的覆盖率
- **忽略列名**,只基于列内容匹配
- **忽略行序**(通过排序实现)
- **支持重复列**(同一列签名需要匹配相同次数)

### 6.3 评分指标计算

评测在 Recall(覆盖率)的基础上引入对**冗余预测**的轻度惩罚,从而更全面地衡量预测结果质量。

**定义:**

- **Matched Columns**:prediction 中成功匹配到 gold 的列数(按列签名匹配)
- **Gold Columns**:gold 中的列总数
- **Predicted Columns**:prediction 中的列总数
- **Extra Columns**:prediction 中未匹配到 gold 的列数

**先计算:**

> Recall = Matched Columns / Gold Columns

**最终得分:**

> Score = Recall − λ · (Extra Columns / Predicted Columns)

其中:

- 惩罚项权重 **λ** 为预设常数,用于平衡覆盖率与冗余预测之间的关系
- 当预测结果包含较多未匹配列时,得分会相应降低
- 最终得分的下界为 **0**

### 6.4 设计动机

该评分方式在保持 Recall 导向的同时**适度约束预测结果的冗余度**,设计目标如下:

- ✅ **鼓励完整覆盖**:优先评估是否找到所有目标列
- ✅ **控制冗余输出**:对包含较多无关列的预测进行惩罚
- ✅ **保持鲁棒性**:不依赖列名或顺序,仅基于数据内容判断

评测不仅关注"是否都找到了"(Recall),还鼓励"尽量简洁准确"的预测结果。

### 6.5 数值与时间标准化规则

在构造列签名前,评测系统会对单元格内容进行标准化,以减少格式差异(如数值与时间类型)的影响。

| 类型           | 标准化规则                                                                                                                       |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **空值**       | 空字符串、`"null"`、`"none"`、`"nan"`、`"nat"`、`"<na>"`(不区分大小写)→ 统一为空字符串 `""`                                    |
| **数值**       | 用 `Decimal` 解析,四舍五入到 **2 位小数**(`ROUND_HALF_UP`)。例:`4200000` → `"4200000.00"`,`0.005` → `"0.01"`                  |
| **日期**       | ISO 8601 格式(`YYYY-MM-DD`)。例:`"2024-3-1"` 必须为 `"2024-03-01"`                                                              |
| **日期时间**   | 带时区:转为 UTC(以 `Z` 结尾);不带时区:保留原始 ISO 格式                                                                       |
| **字符串**     | 去除首尾空白与 `\r\n`;其余原样保留(**区分大小写**)                                                                              |
| **姓名字段**   | 姓+名作为两列,或合并为全名一列,均可接受。例:`"John" + "Smith"` 或 `"John Smith"`                                                |

> ⚠️ **警告**:字符串比较**区分大小写**。例如 `"East Asia"` 与 `"east asia"` 被视为不同的值。请确保预测输出与数据源中的原始字符串格式一致。

### 6.6 评分示例

| `prediction.csv`                          | `gold.csv` | 结果与原因                                                          |
| ----------------------------------------- | ---------- | ------------------------------------------------------------------- |
| A、B、C 列都存在且取值完全匹配             | B、C 列    | ✅ **高分**:覆盖了所有 gold 列;但因多了 A 列而**略低于满分**       |
| B 列存在,缺少 C 列                        | B、C 列    | ⚠️ **低分**:未完整覆盖 gold 列                                      |
| B、C 列存在但 B 取值不匹配                 | B、C 列    | ⚠️ **低分**:部分列匹配失败                                          |
| 包含许多无关列                             | B、C 列    | ⚠️ **大幅降分**:许多冗余列触发惩罚                                  |
| `prediction.csv` 不存在                    | 任意       | ❌ **0 分**:文件缺失                                                |

### 6.7 总分与排行榜

- 每个任务的得分按上述规则**独立计算**
- 总分为所有任务得分的**平均值**
- 排行榜按总分**降序**排列
- 第一阶段(Phase 1)期间,隐藏集划分为 **A 榜**与 **B 榜**两个子集,任务数分别约为 **60** 与 **320**。第一阶段的最终排行榜按与数据比例对齐的权重综合 A 榜与 B 榜的成绩。
- 总分相同的情况下,按**最后一次有效提交时间升序**排列(提交越早排名越靠前)

## 9. 常见问题(FAQ)

**Q1:容器需要自己遍历所有任务吗?**

是的。一个容器处理整个数据集中的所有任务。Agent 需要遍历 `/input` 下的所有 `task_<id>` 目录,对每个任务独立运行推理,并将结果写入对应的 `/output/task_<id>/prediction.csv`。

**Q2:开发期间参赛者可以使用自己的模型吗?**

可以。开发与本地调试阶段,参赛者可以自由使用任意 LLM(OpenAI、Anthropic、本地模型等)。组织方不干预,也不提供开发 API。但代码必须从环境变量读取模型地址与 Key,**不可硬编码**。提交 Docker 镜像后,评测系统会注入指向 Qwen3.5-35B-A3B 的环境变量,代码会自动切换到组织方的统一模型。在官方评测期间,容器内**不得运行任何其他 LLM 作为主要任务求解模型**,包括基于 CPU 的本地 LLM 推理。但是,参赛者仍可在官方硬件限制内使用合适规模的辅助模型(如 embedding 模型)用于文档处理与检索。

**Q3:`prediction.csv` 的列名必须与 `gold.csv` 一致吗?**

不需要。列名不参与评分;评测脚本只比较每列的取值向量(`column_values` 签名)。列名可以任意命名;建议使用语义化名称以便调试。

**Q4:数值精度如何处理?**

评测脚本会将所有数值标准化为 **2 位小数**(四舍五入)。例如,预测值 `4200000` 与标准答案 `4200000.00` 视为相同。建议预测时使用足够的精度(至少 2 位小数)。

**Q5:12 小时超时是单个任务还是所有任务的限制?**

12 小时是**整个容器运行时长的总上限**,包括所有任务的处理时间。建议在代码中为每个任务单独设置超时控制,并在每个任务处理完后立即写出 `<task_id>/prediction.csv`,以避免超时导致已完成结果丢失。

**Q6:`context/` 下的数据文件是否固定?**

**不固定**。不同任务可能包含不同的数据源组合(`csv/`、`db/`、`json/`、`doc/`、`knowledge.md`)。Agent 应**动态检测** `context/` 下实际存在的子目录与文件,而不是假设固定结构。
