# 官方技术规格

## 1. 文档概述

本文档是 KDD Cup 2026 数据代理竞赛的官方技术规范，详细说明了参赛者提交的打包方法、Docker 容器运行时环境、I/O 格式限制、硬件资源限制、模型调用方法和评分机制。

所有参赛队伍在提交解决方案前，必须仔细阅读并严格遵守本文档中的所有要求。评估系统将完全按照本文档中的描述运行容器。任何不符合规范的提交都将导致评估失败或得分为 0 分。

**注：**若本文件与官网其他描述有任何出入，以本文件的最新版本为准。如有任何疑问，请通过Discord或微信群联系组委会。

## 2. 数据格式规范

### 2.1 完整目录结构

在评估过程中，容器内的完整目录结构如下。参与者的代理必须遍历 /input 下的所有任务子目录，按顺序处理它们，并将结果写入相应的 /output/task\_<id>/ 目录。

```
/input/ # 只读挂载，不允许写入
└── task_<id>/ # 每个任务一个子目录，id 是唯一标识符
    ├── task.json # 任务元数据（task_id、难度、问题）
    └── context/ # 任务所需的上下文数据
        ├── csv/ # 可选：结构化 CSV 文件
        ├── db/ # 可选：SQLite 数据库文件
        ├── json/ # 可选：结构化 JSON 文件
        ├── doc/ # 可选：数据文档（Markdown 等）
        └── knowledge.md # 可选：背景知识文档

/output/ # 可读写，参与者写入预测结果
└── task_<id>/ # 与输入任务目录一一对应
    └── prediction.csv # [必需输出] 参与者的预测答案
```

**笔记：**

* context/ 目录下的具体子目录并非每个任务都固定不变；不同的任务可能包含不同的数据源组合。每个任务至少包含 csv/、db/、json/ 和 doc/ 中的一个，并且可能包含 knowledge.md 文件。
* /input 目录以只读方式挂载；任何写入操作都会导致运行时错误。/output 目录由评估系统预先创建；参与者只需写入 task\_<id>/prediction.csv 文件，无需创建父目录。

### 2.2 task.json 格式

每个任务的元数据都存储在 task.json 文件中，字段描述如下：

| 字段名称 | 类型 | 描述 |
| --- | --- | --- |
| 任务 ID | 细绳 | 与 /input/task\_<id>/ 目录名一致的唯一任务标识符 |
| 困难 | 细绳 | 难度等级：简单/中等/困难/极限 |
| 问题 | 细绳 | 描述待解答的数据分析任务的自然语言分析问题 |

task.json 示例：

```
{
  "task_id": "task_11",
  "difficulty": "medium",
  "question": "第三季度哪个产品类别的总收入最高？请列出每个类别及其总收入，并按收入降序排列。"
}
```

### 2.3 背景/数据源描述

context/ 目录下的数据源类型因任务而异。参与者的代理必须检测并读取实际存在的子目录和文件：

| 子目录 | 内容类型 | 描述 |
| --- | --- | --- |
| csv/ | 结构化表格 | 一个或多个 CSV 文件，可直接使用 pandas 和其他工具读取。 |
| 数据库/ | SQLite数据库 | 一个或多个包含多个关系表的 .sqlite / .db 文件 |
| json/ | 结构化数据 | JSON 格式的半结构化数据文件 |
| 文档/ | 数据文档 | Markdown 或其他格式的数据报告和分析文档 |
| knowledge.md | 背景知识 | 与任务相关的背景知识文档，包括业务定义和术语解释。 |

难度与数据来源的对应关系：

| 困难 | 数据模态 | 文件比例 | 核心挑战 |
| --- | --- | --- | --- |
| 简单的 | CSV/JSON + knowledge.md | 短文本 | Python代码生成和数据分析 |
| 中等的 | CSV/JSON + db/ + knowledge.md | 中等的 | 文本转 SQL + 多源数据分析 |
| 难的 | 与上述相同 + 文档/数据文档 | 约 1 万至 12.8 万个代币 | 非结构化文档推理 |
| 极端 | 与困难相同 | >12.8万个代币 | 长上下文工程和内存管理 |

### 2.4 输出格式：prediction.csv

对于每个任务，参与者的代理必须将预测结果写入 /output/task\_<id>/prediction.csv 文件。该文件采用标准 CSV 格式（UTF-8 编码），并满足以下要求：

* 第一行是列名（标题）；列名不参与评分，仅用于提高可读性。
* 从第二行开始是数据行，每一行对应一条结果记录。
* 列顺序不影响评分；评分基于列值向量的无序匹配（参见第 6 节）。
* 数值数据应保持足够的精度；评估系统在进行比较之前会将数值标准化到小数点后两位（四舍五入）。
* 空值应写成空字符串；参与者可以填写 NULL、null、NaN 等，评估系统会将这些值归一化为空字符串。
* 对于姓名栏，将名字和姓氏拆分成两列，或者将它们合并成一列全名，两种形式均可接受；评估系统认为这两种形式都正确。

prediction.csv 示例（对应问题：按类别列出总销售额）：

```
类别，总收入
电子产品，4200000.00
服装，1850000.00
食品，930000.00
```

组织者建议在每次提交前完成以下检查。许多第一轮失败是由于命名、平台、打包、共享权限、输出路径或 CSV 格式问题造成的。

2. 使用 docker save 导出： 构建时`.tar.gz`使用。不要使用（容器导出不是我们流水线可加载的镜像归档）。 `docker save` `docker export`
3. linux/amd64：评估运行在`linux/amd64`. 请确保镜像在该平台上启动（例如，多架构构建或在构建时）。 `--platform=linux/amd64`

6. 有效的 CSV：确认每个文件都是标准的 CSV (UTF-8) 格式，并且可以正确解析，以便评估者可以读取结果。 `prediction.csv`

图片上传后，评估系统将按顺序执行以下步骤：

```

步骤 2：启动评估容器（概念表示，实际参数由评估平台生成）

```
docker run --rm \
  --network=eval_net \ # 内部网络（仅访问评估系统服务）

  --memory=64g \ # 内存硬性限制

  -e MODEL_NAME=<model_name> \ # 内部 LLM 模型访问名称
  team0042:v3 # 参与者解决方案镜像名称

**注意：** /input 和 /output 都挂载整个数据集目录（包含所有任务）。容器必须遍历 /input 下的所有 task\_<id> 子目录并逐个处理它们。

### 3.7 运行时日志规范

评估系统通过 Docker 挂载为每个提交分配一个独立的 /logs 目录（读写权限）。当参赛者的解决方案异常退出时，组织者可以提取 /logs 目录的内容供参赛者调试参考。

**重要提示：**主办方将审核 /logs 目录下的内容。任何包含泄露的测试集数据、黄金答案或其他私人信息的日志都将导致该提交失去调试支持资格；严重违规者将被取消参赛资格。

## 4. 硬件资源限制

### 4.1 资源配额

| 资源类型 | 限度规格 |
| --- | --- |
| 中央处理器 | 16 核（虚拟 CPU），x86-64 架构 |
| 记忆 | 64 GB 内存；如果超出限制，容器将被 OOM 终止，该次提交的所有结果将无效。 |
| GPU | 参与者容器没有 GPU；模型推理由评估系统统一完成（参见第 5 节）。 |
| 运行时限制 | 所有任务的总运行时间限制为 12 小时；容器超时后被强制终止。 |

**注意：** 12 小时是所有任务的总时限，而非单个任务的时限。请合理分配每个任务的处理时间，并在代码中实现超时保护，以确保已完成任务的结果能够及时输出。此外，我们建议参与者使用多线程/多进程来加速不同输入样本的批量并行处理（请参考入门套件代码中的 max\_workers 设置[）](https://github.com/HKUSTDial/kddcup2026-data-agents-starter-kit)。

### 4.2 超时和内存溢出处理

* **超时（12 小时）：** 评估系统发送 SIGTERM 信号并等待 30 秒，如果未退出，则发送 SIGKILL 信号。已写入的 prediction.csv 文件仍将参与评分；未写入的任务得 0 分。
* **OOM Kill：** 容器被强制终止；已写入的 prediction.csv 文件参与评分；未写入的任务得 0 分。
* **非零退出代码：** 不影响评分；评估系统只关心 /output 目录中是否存在文件。

## 5. 模型调用规范

### 5.1 开发阶段与评估阶段

本次竞赛采用“自由开发，统一评价”的模型使用策略：

| 阶段 | 模型来源 | 描述 |
| --- | --- | --- |
| 开发/本地调试 | 参与者选择（任何法学硕士学位） | 主办方不提供开发API；参与者需自行申请和使用各种模型服务。 |
| 正式评估（提交后） | Qwen3.5-35B-A3B（由组织者统一部署） | 评估系统在启动容器时通过环境变量注入模型服务地址；所有参与者使用同一模型以确保公平性。 |

**关键设计原则：** 参赛者代码必须从环境变量中读取模型服务地址和 API 密钥，而不能将其硬编码到镜像中。这样，同一份代码在本地开发时指向参赛者自己的模型，提交后则自动切换到主办方统一部署的 Qwen3.5-35B-A3B 模型。

**公平性约束：** 在官方评估期间，Qwen3.5-35B-A3B 是唯一允许使用的主要任务求解 LLM。参赛者不得在容器内运行任何其他 LLM 作为主要推理或生成引擎，包括 CPU 执行的替代 LLM 的本地推理。在官方硬件限制范围内，允许使用大小合适的辅助模型用于文档处理、检索或表示目的，例如嵌入模型，但前提是这些模型不得用作主要求解器。

启动参与者容器时，评估系统会以 `docker run -e KEY=VALUE` 的形式注入运行时环境变量。参与者程序必须在运行时从环境变量中读取这些配置，不得将 API URL、API 密钥和其他信息硬编码到代码或镜像中。

| 环境变量名称 | 描述 |

| MODEL\_API\_KEY | 模型服务认证密钥（由评估系统注入；参与者无需事先知道具体值） |
| 型号名称 | 评估过程中使用的模型名称，值为“qwen3.5-35b-a3b” |

**⚠️警告：**严禁在代码、配置文件或Docker镜像中硬编码API密钥、API URL和其他敏感配置信息。评估系统启动时注入的环境变量是唯一合法的配置来源。

### 5.3 组织者 vLLM 部署参考

为了帮助参与者将本地测试与官方评估环境保持一致，主办方托管的 Qwen3.5-35B-A3B 服务已通过 vLLM 进行概念部署，使用的参数与以下命令等效。敏感的基础架构字段已故意替换为占位符。在评估期间，参与者仍必须使用 MODEL\_API\_URL、MODEL\_API\_KEY 和 MODEL\_NAME，而不能硬编码端点详细信息。

```
vllm serve <模型路径> \
  --host <主机> \
  --port <端口> \
  --tensor-parallel-size 8 \
  --seed 1024 \
  --served-model-name qwen3.5-35b-a3b \
  --max-model-len 262144 \
  --reasoning-parser qwen3 \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_coder \
  --trust-remote-code
```

| 范围 | 参与者解读 |

## 6. 评分机制

### 6.1 评估过程说明

该评估系统采用统一的自动化评估程序，对参与者提交的预测结果进行离线批量评分。参与者需要提交指定格式（例如 prediction.csv）的结果文件，系统会将这些文件与相应的标准答案进行比对，生成最终得分。

所有提交内容均在相同的评估环境下运行，评估过程采用确定性执行，以确保评估结果的一致性和公平性。

### 6.2 列匹配方法

评估采用列级内容一致性匹配方法（列级匹配），其核心过程如下：

1. 读取 prediction.csv 和 gold.csv 文件，解析列和数据内容
2. 对每一列，对该列中的所有单元格值进行排序，以构建“列签名”。
3. 分别统计预测结果和黄金样本中列特征出现的次数。
4. 基于列特征进行匹配，计算预测对黄金的覆盖率。

* 忽略列名，仅根据列内容进行匹配。
* 忽略行顺序（通过排序实现）
* 支持重复列（同一列的签名需要出现相同次数）

### 6.3 评分指标计算

评估引入了基于覆盖率（召回率）的对冗余预测的轻微惩罚，以更全面地衡量预测结果的质量。

定义：

* **匹配列数：** 预测中成功匹配黄金的列数（按列签名匹配）
* **金柱：** 金柱总数
* **预测列数：** 预测中的列总数
* **额外列：** 预测中与黄金不匹配的列数

首先计算：

召回率 = 匹配列 / 金色列

最终得分是：

得分 = 召回率 - λ · (额外列数 / 预测列数)

在哪里：

* 惩罚项λ权重是一个预设常数，用于平衡覆盖率和冗余预测之间的关系。
* 当预测结果中包含许多不匹配的列时，得分会相应降低。
* 最终得分的下限为 0

### 6.4 设计原理

该评分方法适度限制了预测结果的“冗余程度”，同时保持了召回率导向，其设计目标如下：

* ✅**鼓励全面覆盖：**优先评估是否找到所有目标列。
* ✅**控制冗余输出：**对包含大量无关列的预测结果进行惩罚
* ✅**保持稳健性：**不要依赖列名或列顺序，仅根据数据内容进行判断

评估不仅关注“是否全部找到”（召回率），而且还鼓励“尽可能简洁和准确”的预测结果。

### 6.5 值和时间归一化规则

在构建列签名之前，评估系统会对单元格内容进行规范化，以减少格式差异的影响（例如数值和时间类型的标准化）。

| 类型 | 归一化规则 |
| --- | --- |
| 空值 | 空字符串，"null", "none", "nan", "nat", "<na>"（不区分大小写）→ 统一为空字符串 "" |
| 数字 | 使用 Decimal 解析，四舍五入到小数点后两位（向上取整）。例如，4200000 → "4200000.00", 0.005 → "0.01" |
| 日期 | ISO 8601格式（YYYY-MM-DD）。例如，“2024-3-1”必须写成“2024-03-01”。 |
| 日期时间 | 带时区：转换为 UTC（以 Z 结尾）；不带时区：保持原始 ISO 格式 |
| 细绳 | 删除开头/结尾的空格和 \r\n；其余部分保持不变（区分大小写） |
| 名称字段 | 名字和姓氏可以分两列填写，也可以合并为一个完整姓名写在一列中。例如，“John”+“Smith”或“John Smith”。 |

**⚠️警告：** 字符串比较区分大小写。例如，“East Asia”和“east asia”被视为不同的值。请确保预测输出与数据源中的原始字符串格式一致。

### 6.6 评分示例

| 预测.csv | gold.csv | 结果与原因 |
| --- | --- | --- |
| A、B、C 列均存在，且值完全匹配。 | B、C列 | ✅ 高分：覆盖所有金色列；但由于多了一列 A，所以略低于完美。 |
| B列存在，C列缺失 | B、C列 | ⚠️ 低分：金柱覆盖不完整 |
| B 列和 C 列存在，但 B 列的值不匹配。 | B、C列 | ⚠️ 低分：部分列匹配失败 |
| 包含许多无关的列 | B、C列 | ⚠️ Significantly lower score: many redundant columns incur penalty |
| prediction.csv does not exist | Any | ❌ Score 0: missing file |

### 6.7 Total Score & Leaderboard

* Each task's score is calculated independently according to the above rules
* Total score is the average of all task scores
* Leaderboard is sorted by total score in descending order
* During Phase 1, the hidden set is split into A-board and B-board subsets with approximately 60 and 320 tasks, respectively. The final Phase 1 leaderboard combines A-board and B-board scores using weights aligned with the data proportions.
* If total scores are the same, sort by last valid submission time in ascending order (earlier submission ranks higher)

## 9. 常见问题解答 (FAQ)

### 问题1：容器是否需要自行遍历所有任务？

是的。一个容器处理整个数据集中的所有任务。代理需要遍历 /input 下的所有 task\_<id> 目录，对每个任务独立运行推理，并将结果写入相应的 /output/task\_<id>/prediction.csv 文件。

### Q2：参与者在开发过程中可以使用自己的模型吗？

是的。在开发和本地调试期间，参赛者可以自由使用任何LLM（OpenAI、Anthropic、本地模型等）。主办方不会干预或提供开发API。但是，代码必须从环境变量中读取模型地址和密钥，不能硬编码。提交Docker镜像后，评估系统会注入指向Qwen3.5-35B-A3B的环境变量，代码将自动切换到主办方提供的统一模型。在正式评估期间，容器内不得运行任何其他LLM作为主要任务求解模型，包括基于CPU的本地LLM推理。但是，参赛者仍然可以在官方硬件限制范围内使用适当大小的辅助模型，例如嵌入模型，用于文档处理和检索。

### Q3：prediction.csv 的列名必须与 gold.csv 的列名一致吗？

不。列名不参与评分；评估脚本仅比较每列的值向量（column\_values 签名）。列名可以任意命名；建议使用语义化的名称以便于调试。

### Q4：数值精度是如何处理的？

评估脚本会将所有数值标准化为小数点后两位（四舍五入）。例如，预测值 4200000 和黄金价值 4200000.00 被视为相同。建议在预测时使用足够的精度（至少两位小数）。

### Q5：12 小时超时是指单个任务的超时还是所有任务的超时？

12 小时是整个容器运行的总时长限制，包括所有任务的处理时间。建议在代码中为每个任务设置单独的超时控制，并在处理完每个任务后立即写入 <task\_id>/prediction.csv 文件，以防止超时导致已完成的结果丢失。

### Q6：数据文件是否处于特定上下文/固定状态？

未固定。不同的任务可能包含不同的数据源组合（例如 csv/、db/、json/、doc/、knowledge.md）。代理程序应该动态检测 context/ 目录下实际存在的子目录和文件，而不是假定固定的结构。
