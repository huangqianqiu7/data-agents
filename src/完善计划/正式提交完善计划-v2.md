# 正式提交完善计划（完善版）

本文档在原《正式提交完善计划》基础上合并审查结论，形成 `data_agent_langchain` 面向 KDD Cup 2026 Data Agents 正式 Docker 提交的完整整改计划，原计划文档保持不变。

## 一、当前检查结论

`data_agent_langchain` 已具备 LangGraph 后端、任务遍历、单任务子进程隔离、`prediction.csv` 写出、Tool Calling 能力校验等基础能力；但正式提交还缺少容器入口、官方 `/output` 布局适配、提交态环境变量配置、异常隔离、日志脱敏、`.dockerignore` 密钥隔离和提交态运行预算控制。

已在代码侧核实的关键现状：

| 事项 | 当前状态 | 影响 |
| --- | --- | --- |
| Dockerfile / `.dockerignore` | 仓库根未发现 | 无法直接正式提交 Docker 镜像 |
| 输出路径 | 当前为 `run.output_dir/<run_id>/<task_id>/prediction.csv` | 比官方要求多一层 `<run_id>` |
| 模型配置 | `build_chat_model()` 读取 `config.agent.api_base/api_key/model` | 提交入口必须从 `MODEL_*` 构造配置 |
| Tool Calling caps | `tool_calling` 模式强依赖 `gateway_caps.yaml` | 镜像内缺失会启动失败 |
| LangChain / LangGraph 依赖 | `pyproject.toml` 未显式列出 | Docker 环境可能缺包 |
| runner 批量异常 | ThreadPool `future.result()` 会重抛 | 单题异常可能中断整批任务 |
| 失败任务 CSV | `answer` 不是 dict 时不写 `prediction.csv` | 失败/超时任务文件缺失即 0 分 |
| 日志内容 | trace 可能包含 raw_response 和 observation | 若复制到 `/logs` 有泄漏测试数据风险 |
| LangSmith | 默认 false，但提交入口未强制锁死 | 意外开启会违反网络策略并浪费时间 |

## 二、P0 必须完善项

### 1. Dockerfile / ENTRYPOINT

比赛只接受可运行 Docker 镜像归档。需要新增：

- `Dockerfile`：构建 `linux/amd64` 可运行镜像。
- `ENTRYPOINT`：无需额外参数即可自动处理 `/input` 下全部任务。
- `.dockerignore`：排除密钥、数据、缓存、历史输出和大文件。

ENTRYPOINT 应调用提交专用入口，而不是要求评测系统传入 CLI 参数。

### 2. 官方 `/output` 扁平化

当前内部运行目录可以继续保留：

```text
/tmp/dabench-runs/<run_id>/<task_id>/prediction.csv
```

但最终必须写成：

```text
/output/<task_id>/prediction.csv
```

提交入口需要在每题完成后立即同步或复制对应 `prediction.csv`，不能等所有任务结束后才统一复制，以降低 12 小时超时或容器被杀导致结果丢失的风险。

### 3. `MODEL_*` 环境变量注入，并且提交态不读 YAML

正式提交入口不应加载本地 YAML，避免误读含密钥配置。建议直接构造 `AppConfig`：

```python
agent = AgentConfig(
    model=os.environ.get("MODEL_NAME", "qwen3.5-35b-a3b"),
    api_base=os.environ["MODEL_API_URL"],
    api_key=os.environ.get("MODEL_API_KEY", "EMPTY"),
    action_mode="tool_calling",
)
```

约束：

- `MODEL_API_URL` 缺失时立即失败并写 `/logs/runtime.log`。
- `MODEL_API_KEY` 缺失时允许回退 `EMPTY`，兼容 vLLM 常见测试方式。
- `MODEL_NAME` 缺失时回退 `qwen3.5-35b-a3b`。
- 不把任何真实 URL / Key 写进代码、配置、Dockerfile 或日志。

### 4. per-task 异常隔离

规则 §4.2 明确：已写出的结果仍参与评分，未写出任务为 0 分。当前 `run_benchmark()` 的 ThreadPool 分支中 `future.result()` 会重抛异常，一题 schema 偏差或加载失败可能中断整批任务。

提交入口必须保证：

- 每个 `task_<id>` 独立 try/except。
- 单题失败只记录到 `/logs/failed_tasks.jsonl`，不中断其他题。
- 错误摘要只包含 `task_id`、错误类型、错误首行、耗时，不包含上下文数据和 LLM 输出。

### 5. 失败任务 `prediction.csv` 兜底

当前 `_write_task_outputs()` 在 `answer` 不是 dict 时不写 CSV。提交态建议在每题开始前先创建可解析占位文件：

```text
/output/<task_id>/prediction.csv
```

占位 CSV 至少应是标准 UTF-8 CSV。成功后用真实答案覆盖。这样即使任务超时、异常或容器收到 SIGTERM，输出目录里也尽量保留可解析文件。

需要注意：占位 CSV 不能伪造答案；它只是降低文件缺失导致的额外风险。

### 6. `/logs` 内容审计与脱敏

规则 §3.7 警告日志包含测试集数据、标准答案或私有信息会导致调试支持资格丧失，严重者取消参赛资格。当前 trace 可能包含：

- LLM `raw_response`
- 工具 observation
- CSV / JSON / Markdown 数据片段

提交态约束：

- `/logs/runtime.log` 只记录任务状态、错误类型、错误首行、步数、耗时。
- `trace.json` 留在内部 run 目录，不复制到 `/logs`。
- `/logs/failed_tasks.jsonl` 不记录 observation、raw_response、完整 prompt、上下文片段或 API key。
- 入口打印环境变量时只允许打印是否存在，不打印值。

### 7. `.dockerignore` 密钥隔离

`.dockerignore` 至少应包含：

```text
.env
.git/
configs/
data/
old/
Comparison/
artifacts/runs/
artifacts/dabench-lc/
.pytest_cache/
**/__pycache__/
**/.ipynb_checkpoints/
```

注意：如果正式 `tool_calling` 依赖 `artifacts/gateway_caps.yaml`，不能简单排除整个 `artifacts/`。可选方案：

- `.dockerignore` 排除 `artifacts/runs/`，但保留 `artifacts/gateway_caps.yaml`。
- 或把提交态 caps 文件放到镜像固定路径，例如 `/app/gateway_caps.yaml`，并在 `AppConfig` 中覆盖 `observability.gateway_caps_path`。

### 8. LangSmith 提交态锁死为 false

提交入口应强制：

- `observability.langsmith_enabled=False`
- 清理 `LANGSMITH_*`、`LANGCHAIN_API_KEY`、`LANGCHAIN_TRACING_V2` 等环境变量

目的：避免正式评测时尝试连接外部 LangSmith，违反 §5.5 网络策略或拖慢 12 小时总预算。

## 三、P1 建议完善项

### 1. 放宽 `task.json` schema 校验

当前 `_load_task_record` 严格要求键集完全等于 `{task_id, difficulty, question}`。如果官方未来在 `task.json` 增加 `language`、`version`、`tags` 等字段，会触发异常。

建议改为：

- 必须包含 `task_id`、`difficulty`、`question`。
- 多余字段忽略。
- 缺少必需字段才报错。

该项不属于 Docker 化，但与正式提交鲁棒性强相关。

### 2. `gateway_caps.yaml` 提交态路径明确化

当前默认路径是：

```text
<PROJECT_ROOT>/artifacts/gateway_caps.yaml
```

正式提交需要明确采用以下一种策略：

- 保持项目布局，在镜像内提供 `/app/artifacts/gateway_caps.yaml`。
- 或复制到 `/app/gateway_caps.yaml`，提交入口构造 `AppConfig` 时覆盖 `observability.gateway_caps_path`。

建议采用第二种，减少与开发 artifacts 目录耦合。

### 3. 总运行时预算分档

规则限制整个容器最多 12 小时。若 B 榜约 320 题，当前默认：

```text
320 题 × 600s ÷ 4 worker = 13.3h
```

已超过 12 小时，未计冷启动、模型等待和清理时间。建议：

| 场景 | max_workers | task_timeout_seconds | 说明 |
| --- | ---: | ---: | --- |
| A 榜约 60 题 | 4 | 600 | 当前默认基本可用 |
| B 榜约 320 题 | 8 | 480 | 理论上限约 5.3h，留足缓冲 |
| 保守低资源 | 4 | 360 | 更稳但可能牺牲难题求解时间 |

同时提交入口设置全局软截止：运行到第 11 小时后停止派发新任务，等待 in-flight 任务收尾并确保已完成结果落盘。

### 4. 显式固化 LangChain / LangGraph 依赖

`pyproject.toml` 应显式包含当前源码实际依赖，例如：

- `langchain-core`
- `langchain-openai`
- `langgraph`

具体版本应以当前 `dataagent` 环境中验证通过的版本为准，避免 Docker 重新解析到不兼容版本。

### 5. `None` 写 CSV 的行为测试

规则 §6.5 会把空字符串、`null`、`nan` 等归一为空。当前 `csv.writer` 通常会把 Python `None` 写为空 cell，但需要测试确保 finalize 没有提前把 `None` 转成字符串 `"None"`。

建议新增测试：`answer.rows` 中的 `None` 落盘后必须为空 cell。

## 四、P2 清理项

### 1. CLI 过期文案同步

`gateway-smoke` 末尾仍输出默认 action mode 保持 `json_action` 的旧文案，但当前默认已是 `tool_calling`。建议同步删除或改为当前状态说明。

## 五、推荐实施顺序

### 第 1 步：测试先行

新增提交入口相关测试，覆盖：

- 缺少 `MODEL_API_URL` 时失败并给出明确错误。
- `MODEL_API_KEY` 缺失时回退 `EMPTY`。
- `MODEL_NAME` 缺失时回退 `qwen3.5-35b-a3b`。
- 构造出的 `AppConfig.dataset.root_path == /input`。
- 构造出的 `AppConfig.run.output_dir` 指向内部目录，不直接污染 `/output`。
- 构造出的 `AppConfig.observability.langsmith_enabled is False`。
- 内部结果扁平复制到 `/output/<task_id>/prediction.csv`。
- 不产生 `/output/<run_id>/<task_id>/prediction.csv`。
- 失败任务写 `/logs/failed_tasks.jsonl`，且不包含 observation/raw_response/API key。
- `None` 落盘为空 cell。

### 第 2 步：新增 `data_agent_langchain.submission`

模块职责：

1. 从 `MODEL_*` 构造 `AppConfig`。
2. 校验并创建 `/output`、`/logs`。
3. 清理 LangSmith 相关环境变量。
4. 设置 `gateway_caps_path`。
5. 对每个任务单独运行、单独捕获异常。
6. 每题开始先写占位 `prediction.csv`。
7. 成功后覆盖真实 `prediction.csv`。
8. 只把脱敏运行摘要写入 `/logs`。

### 第 3 步：补 Docker 提交层

新增：

- `Dockerfile`
- `.dockerignore`
- `gateway_caps.yaml` 提交态文件或 Docker 构建时复制规则

Dockerfile 要求：

- 支持 `linux/amd64` 构建。
- 安装项目依赖。
- 设置默认 `ENTRYPOINT`。
- 不复制 `.env`、`configs/`、本地数据和 artifacts 历史输出。

### 第 4 步：增强数据集 schema 鲁棒性

修改 `benchmark/dataset.py`：

- `task.json` 必含三键。
- 忽略多余字段。
- 保持缺字段时报错。

### 第 5 步：运行验证

必须运行：

```text
C:\Users\18155\anaconda3\envs\dataagent\python.exe -m pytest tests/ -q
```

建议补充：

```text
C:\Users\18155\anaconda3\envs\dataagent\python.exe -m data_agent_langchain.cli --help
```

容器级 smoke：

```text
docker build --platform=linux/amd64 -t <team_id>:v<N> .

docker run --rm \
  -v <local_input>:/input:ro \
  -v <local_output>:/output:rw \
  -v <local_logs>:/logs:rw \
  -e MODEL_API_URL=<模型服务地址> \
  -e MODEL_API_KEY=<密钥或 EMPTY> \
  -e MODEL_NAME=qwen3.5-35b-a3b \
  <team_id>:v<N>
```

验证重点：

- `/output` 下只有 `task_<id>/prediction.csv`。
- 不存在 `/output/<run_id>/...`。
- `prediction.csv` 可被标准 CSV 解析。
- `/logs/runtime.log` 存在。
- `/logs/failed_tasks.jsonl` 不包含上下文数据、LLM 原始输出或密钥。
- 容器无需交互输入。

## 六、正式提交检查清单

- 镜像名符合 `<team_id>:v<N>`。
- 归档名符合 `<team_id>_v<N>.tar.gz`。
- 使用 `docker save` 导出，不使用 `docker export`。
- 镜像支持 `linux/amd64`。
- 归档大小小于 10 GB。
- 镜像内不包含 `.env`、本地 API key、`configs/react_baseline.example.yaml` 或其他敏感 YAML。
- 代码不硬编码真实 `MODEL_API_URL`、`MODEL_API_KEY`、`MODEL_NAME`。
- 容器只访问 `MODEL_API_URL` 指定的内部模型服务。
- 容器不修改 `/input`。
- 输出路径为 `/output/<task_id>/prediction.csv`。
- 每题失败不影响其他题继续运行。
- `/logs` 只包含脱敏日志和错误摘要。
- `langsmith_enabled` 在提交态强制为 false。
- `gateway_caps.yaml` 路径在镜像内可用。
- Google Drive 链接权限为 Anyone with the link / Viewer。
- 提交邮件从队长注册邮箱发送。

## 七、修订优先级表

| 优先级 | 事项 | 说明 |
| --- | --- | --- |
| P0 | Dockerfile + ENTRYPOINT | 没有它无法正式提交 |
| P0 | `/output/<task_id>/prediction.csv` 扁平化 | 路径不合规会直接影响评分 |
| P0 | `MODEL_*` 注入 + 不读 YAML | 避免硬编码和敏感配置泄漏 |
| P0 | `.dockerignore` 排除 `.env` / `configs/` / `data/` / 历史 artifacts | 防止密钥和大文件进入镜像 |
| P0 | per-task try/except | 防止单题异常中断整批 |
| P0 | 失败任务占位 CSV | 减少文件缺失风险 |
| P0 | `/logs` 脱敏策略 | 避免泄漏测试集数据或 LLM 原始输出 |
| P0 | LangSmith 提交态锁死 false | 符合网络策略并避免外联阻塞 |
| P1 | `task.json` schema 放宽 | 容忍官方新增字段 |
| P1 | gateway caps 文件入镜像并明确路径 | `tool_calling` 默认模式依赖它 |
| P1 | A/B 榜运行预算分档 | 避免 320 题超 12 小时 |
| P1 | 显式固化 LangChain / LangGraph 依赖 | 避免 Docker 缺包或版本漂移 |

| P2 | CLI 过期文案修正 | 降低维护误导 |

## 八、最终建议

继续采用“新增最小提交适配层 `data_agent_langchain.submission`，不大改现有 runner”的方向，但该提交层不能只做输出复制；它还必须承担提交态配置构造、per-task 异常隔离、失败 CSV 兜底、日志脱敏、LangSmith 禁用和 caps 路径固定。完成这些 P0 项后，再编写 Dockerfile 与容器级 smoke，正式提交风险会明显降低。
