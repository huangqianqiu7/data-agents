# 正式提交完善计划 v4

本文档基于 v3 计划、v3 审查结论、`rules.zh.md` 与 `src/data_agent_langchain` 当前代码核对结果，形成可直接进入实现阶段的 KDD Cup 正式 Docker 提交完善计划。

## 一、当前代码核对结论

`data_agent_langchain` 已具备 LangGraph 后端、任务加载、子进程隔离、超时控制、CSV 写出、Tool Calling 绑定和基础可观测能力。但正式提交仍需要补齐提交态入口、容器层、输出布局、异常隔离、日志脱敏、依赖减法瘦身和容器信号处理。

| 事项 | 当前状态 | v4 判断 |
| --- | --- | --- |
| Dockerfile / `.dockerignore` | 仓库根未提供提交态容器文件 | P0 |
| 输出路径 | runner 写到 `<run_id>/<task_id>/prediction.csv` | P0，需扁平化到 `/output/<task_id>/prediction.csv` |
| 模型配置 | `build_chat_model()` 使用 `config.agent.model/api_base/api_key` | P0，提交入口直接从 `MODEL_*` 构造 `AppConfig` |
| Tool Calling caps | `runner.py` 在 `tool_calling` 默认模式下强校验 `gateway_caps.yaml` | P0，缺文件会全批启动失败 |
| 批量异常 | `future.result()` 会重抛异常 | P0，提交态需 per-task try/except |
| 失败 CSV | `answer` 非 dict 时不写 `prediction.csv` | P0，需占位 CSV 兜底 |
| 日志 | trace 可能包含 raw_response / observation | P0，`/logs` 只允许脱敏摘要 |
| LangSmith | `tracer.py` 已有双重保护，默认 false | P0 测试守护，提交入口显式锁死 false |
| `task.json` schema | 当前要求键集完全等于三键 | P1，需容忍官方新增字段 |
| 依赖 | 主依赖含 legacy baseline 包，未列 LangChain / LangGraph | P0，需减法重构主依赖 |
| `dabench` entry point | 指向已归档 baseline 包 | P2，建议删除避免误用 |

## 二、P0 必须完成项

### 1. Dockerfile + Python ENTRYPOINT

新增提交态 Docker 层：

- `Dockerfile`
- `.dockerignore`
- 默认 `ENTRYPOINT`
- 镜像内固定 `gateway_caps.yaml`

Dockerfile base image 明确为：

```dockerfile
FROM --platform=linux/amd64 python:3.10-slim
```

理由：

- 与 `pyproject.toml` 的 `requires-python = ">=3.10"` 兼容。
- 比完整 Python 镜像更小。
- 避免 Alpine/musl 对 `numpy`、`pandas` wheels 的兼容风险。

ENTRYPOINT 必须调用：

```text
python -m data_agent_langchain.submission
```

日志必须由 Python `logging.FileHandler('/logs/runtime.log')` 控制写入内容。禁止在 shell ENTRYPOINT 中使用：

```text
2>&1 | tee /logs/runtime.log
```

原因：`tee` 会把所有 stdout/stderr 原样写入日志，可能包含工具 observation、LLM raw_response 或上下文片段，违反日志脱敏目标。

### 2. `gateway_caps.yaml` 固定入镜像

当前 `AgentConfig.action_mode` 默认是 `tool_calling`，runner 会强制读取 `config.observability.gateway_caps_path`。如果镜像内缺少 caps 文件或 `tool_calling=false`，容器启动即失败，所有任务为 0。

v4 固定策略：

- 在仓库新增提交态文件：`docker/gateway_caps.yaml`。
- Dockerfile 显式复制到：`/app/gateway_caps.yaml`。
- 提交入口构造 `AppConfig` 时设置：`gateway_caps_path=Path('/app/gateway_caps.yaml')`。
- 不依赖开发态 `artifacts/gateway_caps.yaml`，避免被 smoke 过程覆盖。

`docker/gateway_caps.yaml` 字面内容：

```yaml
gateway_caps:
  tool_calling: true
  parallel_tool_calls: true
  seed_param: true
  strict_mode: true
```

Dockerfile 复制点：

```dockerfile
COPY docker/gateway_caps.yaml /app/gateway_caps.yaml
```

单测覆盖：文件不存在时给出明确错误；文件存在且 `tool_calling=true` 时可继续。

### 3. 官方 `/output` 扁平化与同目录原子覆盖

内部运行目录可继续保留：

```text
/tmp/dabench-runs/<run_id>/<task_id>/prediction.csv
```

但评测输出必须是：

```text
/output/<task_id>/prediction.csv
```

提交入口要求：

- 每题开始前创建 `/output/<task_id>/prediction.csv` 占位文件。
- 成功后将真实答案覆盖到官方输出路径。
- 不产生 `/output/<run_id>/...`。

占位 CSV 明确为：

```python
PLACEHOLDER_CSV = "result\n"
```

真实答案覆盖必须避免跨挂载点 `os.replace()`。`/tmp/dabench-runs` 与 `/output` 可能是不同设备，直接 `os.replace('/tmp/...', '/output/...')` 会触发 `EXDEV`。

正确流程：

```python
target = Path(f"/output/{task_id}/prediction.csv")
tmp = target.with_name(".prediction.csv.tmp")
shutil.copy(internal_csv, tmp)
os.replace(tmp, target)
```

要求：临时文件必须与最终 `prediction.csv` 在同一目录下，确保最后一步是同设备原子重命名。

### 4. `MODEL_*` 环境变量注入，提交态不读 YAML

提交入口不调用 `load_app_config()`，避免读取本地 YAML 或敏感配置。直接构造核心配置，并显式固定 `graph_mode="plan_solve"`：

```python
dataset = DatasetConfig(root_path=Path("/input"))
run = RunConfig(
    output_dir=Path("/tmp/dabench-runs"),
    max_workers=int(os.environ.get("DABENCH_MAX_WORKERS", 8)),
    task_timeout_seconds=int(os.environ.get("DABENCH_TASK_TIMEOUT_SECONDS", 480)),
)
observability = ObservabilityConfig(
    langsmith_enabled=False,
    gateway_caps_path=Path("/app/gateway_caps.yaml"),
)
agent = AgentConfig(
    model=os.environ.get("MODEL_NAME", "qwen3.5-35b-a3b"),
    api_base=os.environ["MODEL_API_URL"],
    api_key=os.environ.get("MODEL_API_KEY", "EMPTY"),
    action_mode="tool_calling",
)
graph_mode = "plan_solve"
```

约束：

- `MODEL_API_URL` 缺失时立即失败，并只写脱敏错误到 `/logs/runtime.log`。
- `MODEL_API_KEY` 缺失时回退 `EMPTY`。
- `MODEL_NAME` 缺失时回退 `qwen3.5-35b-a3b`。
- 不打印环境变量值，只记录是否存在。
- 只有后续基准验证发现 `plan_solve` 因调用过多导致集中超时时，才考虑切回 `react`。

### 5. per-task 异常隔离与 SIGTERM 处理

正式提交入口不能直接依赖 `run_benchmark()` 的整体 ThreadPool 流程，因为 `future.result()` 重抛异常时可能中断整批任务。

提交态策略：

- 入口先列出 `/input` 下所有 `task_*`。
- 每个任务单独执行 `run_single_task()`。
- 单题 try/except，失败只影响该任务。
- 每题失败写一行 `/logs/failed_tasks.jsonl`。
- 错误摘要只包含：`task_id`、错误类型、错误首行、耗时。
- 不写 observation、raw_response、prompt、上下文片段或密钥。

SIGTERM 处理：

- 主进程注册 `signal.SIGTERM` handler，设置 `_shutting_down=True`。
- 任务派发循环在提交新任务前检查 `_shutting_down`，为 true 时停止派发。
- `ThreadPoolExecutor.shutdown(wait=True, cancel_futures=False)` 等待 in-flight 任务自然结束。
- 依赖 runner 现有 `_reap_process()` 对子进程做 terminate/kill 兜底。

该处理与规则 §4.2 的“已写入结果仍参与评分”语义一致，可尽量保住已完成任务输出。

### 6. `/logs` 脱敏与 summary schema

`/logs` 仅用于排错，不应复制完整 trace。

允许写入：

- `/logs/runtime.log`
- `/logs/failed_tasks.jsonl`
- `/logs/submission_summary.json`

禁止写入：

- `trace.json`
- LLM raw_response
- 工具 observation
- 上下文文件内容片段
- prompt 全文
- API URL / API Key 的实际值
- 失败堆栈全文

`/logs/submission_summary.json` 最小 schema：

```json
{
  "task_count": 60,
  "succeeded_count": 55,
  "failed_count": 5,
  "elapsed_seconds": 7234.2,
  "started_at": "2026-05-08T03:00:00Z",
  "finished_at": "2026-05-08T05:00:34Z",
  "max_workers": 8,
  "task_timeout_seconds": 480,
  "graph_mode": "plan_solve"
}
```

明确不包含：

- model name 与 api_base 的实际值。
- task 级 trace / observation / answer 内容。
- 失败堆栈全文。

测试必须包含：mock observation、raw_response 和明显 API key 字符串后，断言 `/logs` 下所有文件均不包含这些字符串。

### 8. 依赖减法瘦身与 entry point 清理

v3 的 `submission` extras 方案是加法：`pip install ".[submission]"` 仍会安装主 `dependencies` 的所有 legacy 包，因此不能达到瘦身目的。

v4 正确策略是修改主 `dependencies` 为 `data_agent_langchain` 实际运行最小集：

```toml
dependencies = [
    "langchain-core",
    "langchain-openai",
    "langgraph",
    "openai>=2.28.0",
    "pandas>=2.3.3",
    "pydantic>=2.10.6",
    "pyyaml>=6.0.2",
    "typer>=0.15.2",
    "numpy>=2.2.6",
]
```

把 baseline 专属依赖移到 optional dependency：

```toml
[project.optional-dependencies]
baseline = [
    "datasets>=3.3.2",
    "duckdb>=1.5.0",
    "huggingface-hub>=0.29.3",
    "openpyxl>=3.1.5",
    "polars>=1.39.0",
    "pyarrow>=23.0.1",
    "rich>=13.9.4",
]
dev = [
    "pytest>=8.3.5",
    "ruff>=0.11.0",
]
```

Dockerfile 使用：

```text
pip install --no-cache-dir .
```

不带 extras。

同时建议删除过期 entry point：

```toml
dabench = "data_agent_baseline.cli:main"
```

只保留：

```toml
dabench-lc = "data_agent_langchain.cli:main"
```

理由：`data_agent_baseline` 已归档到 `src/废弃/`，该 entry point 被调用时会 `ImportError`，容易误导提交态排错。

### 9. LangSmith 提交态显式关闭

`observability/tracer.py` 已有双重保护：`langsmith_enabled=False` 或缺少 `LANGSMITH_API_KEY` / `LANGCHAIN_API_KEY` 时返回空 callback。v4 仍要求提交入口显式关闭并测试守护：

- `observability.langsmith_enabled=False`
- 清理 `LANGSMITH_*`、`LANGCHAIN_API_KEY`、`LANGCHAIN_TRACING_V2`

这是合规声明与防回归措施，不是当前代码的直接缺陷。

## 三、P1 建议完善项

### 1. 放宽 `task.json` schema

当前 `_load_task_record()` 要求键集完全等于：

```text
task_id, difficulty, question
```

建议改为：

- 必须包含三键。
- 多余字段忽略。
- 缺少必需字段才抛 `DatasetError`。

该项提高对官方数据集字段扩展的鲁棒性。

### 2. A/B 榜运行预算分档

| 场景 | max_workers | task_timeout_seconds | 说明 |
| --- | ---: | ---: | --- |
| A 榜约 60 题 | 4 | 600 | 当前默认基本可用 |
| B 榜约 320 题 | 8 | 480 | 理论上限约 5.3h，留冷启动和失败缓冲 |

提交入口还应设置全局软截止：运行到第 11 小时后停止派发新任务，等待 in-flight 任务收尾并确保已完成结果落盘。

### 3. `None` 写 CSV 行为测试

规则会把空值统一处理，但不能把 Python `None` 写成字符串 `"None"`。新增测试确保：

- `answer.rows` 中的 `None` 写盘为空 cell。
- 空 cell 能被标准 CSV reader / pandas 正常解析。

### 4. 密钥全文件泄漏防回归测试

mock 一个明显的 API key 字符串，例如：

```text
sk-MOCK-DO-NOT-LEAK-12345
```

注入 `MODEL_API_KEY` 后运行最小任务，断言以下路径下所有文本文件都不包含该字符串：

- `/logs/`
- `/output/`
- 内部 `/tmp/dabench-runs/summary.json`
- 内部 `/tmp/dabench-runs/**/metrics.json`

如果 trace 保留在内部 run 目录，也应确认 trace 不包含 key；若无法保证，提交态应避免将 key 写入 run_result 或 trace payload。

### 5. CLI 过期文案修正

`gateway-smoke` 末尾仍有旧文案，建议同步为当前 `tool_calling` 默认状态。该项为 P2 清理，不阻塞正式提交。

## 五、正式提交检查清单

- 镜像名符合 `<team_id>:v<N>`。
- 归档名符合 `<team_id>_v<N>.tar.gz`。
- 使用 `docker save` 导出，不使用 `docker export`。
- 镜像支持 `linux/amd64`。
- 镜像建议小于 2 GB，必须小于 10 GB。
- 镜像基于 `python:3.10-slim` 或经验证兼容的 slim Python 镜像。
- 镜像内不包含 `.env`、本地 API key、`configs/react_baseline.example.yaml` 或其他敏感 YAML。
- 镜像内包含 `/app/gateway_caps.yaml`，且 `tool_calling=true`。
- 代码不硬编码真实 `MODEL_API_URL`、`MODEL_API_KEY`、`MODEL_NAME`。
- 容器只访问 `MODEL_API_URL` 指定的内部模型服务。
- 容器不修改 `/input`。
- 输出路径为 `/output/<task_id>/prediction.csv`。
- 成功覆盖使用 `/output/<task_id>/.prediction.csv.tmp` 同目录临时文件。
- 每个任务失败不影响其他任务继续运行。
- 失败任务仍有可解析占位 CSV。
- `/logs` 只包含脱敏日志、失败摘要和 summary。
- `runtime.log` 不包含 observation、raw_response、prompt、上下文片段或密钥。
- `submission_summary.json` 不包含模型 URL、API key、trace、answer 内容或失败堆栈全文。
- `langsmith_enabled` 在提交态强制为 false。
- 主依赖不包含 legacy baseline 专属大包。
- 容器无需交互输入。
- Google Drive 链接权限为 Anyone with the link / Viewer。
- 提交邮件从队长注册邮箱发送。

## 六、v4 优先级表

| 优先级 | 事项 | 说明 |
| --- | --- | --- |
| P0 | Dockerfile + Python ENTRYPOINT | 无容器入口无法提交 |
| P0 | `/app/gateway_caps.yaml` 固定入镜像并给出字面内容 | `tool_calling` 默认模式启动强依赖 |
| P0 | `/output/<task_id>/prediction.csv` 扁平化 | 官方评分路径要求 |
| P0 | 占位 CSV `result\n` + 同目录原子覆盖 | 避免失败任务文件缺失、半写与跨设备 `EXDEV` |
| P0 | `MODEL_*` 注入 + 不读 YAML | 避免敏感配置泄漏 |
