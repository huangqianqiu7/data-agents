## 二、v2 仍需修的实质问题

### 1. `gateway_caps.yaml` 应升 P0（v2 放在 P1）

`runner.py:193-200` 在 `action_mode='tool_calling'`（默认）时强校验 `gateway_caps.yaml` 必须存在且 `tool_calling=true`。镜像内若该文件缺失，容器启动即抛 `ConfigError`，整批任务全部 0 分，比单题超时严重得多。属于「不工作就完全失败」的 P0，而非「已工作但需优化」的 P1。

**建议改动**：

- 在 §二 加一条 P0 项「`gateway_caps.yaml` 必须复制进镜像，提交入口固定 `gateway_caps_path=/app/gateway_caps.yaml`」。
- §一 状态表里 Tool Calling caps 那一行也要同步标 P0。

### 2. `.dockerignore` 列表仍有遗漏

实际仓库扫描发现 v2 列表外还应包含：

| 路径 | 原因 |
| --- | --- |
| `tests/` | 24 个测试文件 + fixtures，约数 MB |
| `test_api.py` / `test_bench_comparison.py` / `evaluation_report.csv`（仓库根） | 本地 / 历史脚本 |
| `src/test_react_parsers.py` | `src/` 下散落的测试脚本 |
| `src/废弃/` | legacy `data_agent_common` / `data_agent_refactored` / `data_agent_baseline` 全部归档于此，不进 langchain 运行时 |
| `src/LangChain/` | 迁移文档目录，非运行时 |
| `__MACOSX/` | macOS 解压垃圾 |
| `*.tar.gz` / `*.tar` | 防止误把镜像归档复制进新镜像 |
| `.vscode/` | IDE 配置 |
| `*.bak` / `*.swp` / `*.tmp` | 编辑器临时文件 |

**建议改动**：在 §二.7 末尾加一段：

> 最终 `.dockerignore` 应通过 `docker build --no-cache --progress=plain` 检查 `COPY` 步骤产生的层体积，目标镜像 < 2 GB（远低于 10 GB 限制）。如果某层异常大，回查未排除的目录。

### 3. 占位 CSV 的具体内容未指明

§二.5 写「至少应是标准 UTF-8 CSV」但没给字节内容。0 字节文件会让 `pandas.read_csv` 抛 `EmptyDataError`，与「文件不存在」等价。建议明确：

```python
PLACEHOLDER_CSV = "result\n"
```

并明确「真实答案写盘是原子覆盖（写到临时文件再 `os.replace`）而非 append」，避免 worker 异常时留下半截写入的 CSV。

### 4. ENTRYPOINT 必须是 Python logging，不能 `tee`

规则 §3.7 推荐 `python main.py 2>&1 | tee /logs/runtime.log`，但若用 `tee`，**所有 stdout 包括工具 observation / LLM raw_response 会全部进 `runtime.log`**，违反 §二.6 的脱敏目标。

v2 §五.5 容器 smoke 命令直接 `<team_id>:v<N>`，没显示 ENTRYPOINT 写法。建议在 §二.1 明确：

> ENTRYPOINT 调用 `python -m data_agent_langchain.submission`，由提交入口用 `logging.FileHandler('/logs/runtime.log')` 控制写入内容；禁止在 ENTRYPOINT shell 脚本里用 `2>&1 | tee` 简单转发 stdout。

并在测试列表（§五.1）加一条：「mock LLM 回放真实 observation 字符串后，`/logs/runtime.log` 不出现该字符串子串」。

### 5. 依赖固化策略未决：`pyproject.toml` 还是 `uv.lock`

仓库根有 `uv.lock`（511 KB）。v2 §三.4 只说「显式列 langchain-core / langchain-openai / langgraph，版本以 dataagent 环境验证为准」，但没说 Dockerfile 用哪种安装方式：

- 方案 A：`pip install -e .` 走 `pyproject.toml` 解析（版本漂移风险）
- 方案 B：`uv pip sync uv.lock` 锁版本

更关键：`pyproject.toml` 当前列了 `datasets>=3.3.2` / `huggingface-hub>=0.29.3` / `polars` / `pyarrow` / `duckdb` / `openpyxl`，但 `data_agent_langchain` 源码里没有实际 `import huggingface` 或 `import datasets`（已 grep 验证）。这些是 legacy `data_agent_baseline` 的依赖，会让镜像膨胀几百 MB。

**建议改动**：

- 在 `pyproject.toml` 新增 optional dependency 组 `submission`，只列运行时真正需要的：
  - `langchain-core`、`langchain-openai`、`langgraph`
  - `openai`、`pandas`、`pyyaml`、`typer`、`pydantic`、`numpy`
- Dockerfile 用 `pip install ".[submission]"`，跳过 baseline 的 `datasets` / `huggingface-hub` 系列。
- 该项可在 §三.4 升级或在 §二 加 P0「精简提交态依赖」。

## 三、内部一致性 / 小修

### 1. §一 状态表与 §二 优先级错位

§一 表格里「Tool Calling caps」列「`tool_calling` 模式强依赖 `gateway_caps.yaml`」，但对应的整改写在 §三.2 P1。按 §二.1 升级建议成立后，§一 与 §二 应同步加一行。

### 2. §三.3 表格的「保守低资源」一档可删

A 榜 60 题 + B 榜 320 题两档已覆盖；第三档 4w/360s 没明确触发条件，反而徒增决策成本。

### 3. §五.5 docker run smoke 的 task 来源未指明

直接挂 `<local_input>:/input:ro`，但没说 `<local_input>` 应该是 fake 任务还是公共 DABench 集合。建议明确：

> 用 `tests/fixtures/sample_tasks/`（如不存在则新建）下放 1-2 个最小 task；不要直接挂当前仓库的 `data/` 公共集合，避免依赖大数据 + 避免 LLM 实际跑很久。

### 4. §二.3 代码片段可补全 6 个字段

当前只展示了 `agent` 部分。完整版本：

```python
dataset = DatasetConfig(root_path=Path("/input"))
run = RunConfig(
    output_dir=Path("/tmp/dabench-runs"),
    max_workers=int(os.environ.get("DABENCH_MAX_WORKERS", 8)),
    task_timeout_seconds=int(os.environ.get("DABENCH_TASK_TIMEOUT_SECONDS", 480)),
)
observability = ObservabilityConfig(
    langsmith_enabled=False,
    gateway_caps_path=Path("/app/gateway_caps.yaml"),
)
agent = AgentConfig(
    model=os.environ.get("MODEL_NAME", "qwen3.5-35b-a3b"),
    api_base=os.environ["MODEL_API_URL"],
    api_key=os.environ.get("MODEL_API_KEY", "EMPTY"),
    action_mode="tool_calling",
)
```

让构造 `AppConfig` 的关键字段在一段代码里完整呈现。

### 5. tracer.py 已是 belt+suspenders（信息更新）

`observability/tracer.py:28-31` 本身就双重判断 `langsmith_enabled` 与 `LANGSMITH_API_KEY` / `LANGCHAIN_API_KEY` 环境变量，任一不满足即返回 `[]`。所以 §二.8 的 LangSmith 锁死实际是显式化已有保护，不是修缺陷。表述上可以少一点危机感、多强调「显式声明 + 测试守护」。

## 四、综合判断

v2 已经从「容器化路径 + I/O」覆盖到「合规与异常隔离」全栈。剩下 5 个问题里：

- **必须改的**：§二.1（gateway caps 升 P0）、§二.3（占位 CSV 字节内容）、§二.4（ENTRYPOINT 写法）。
- **强烈建议**：§二.2（dockerignore 补漏）、§二.5（依赖瘦身）。

修完这 5 项就可以正式开工 Dockerfile + `data_agent_langchain.submission` 模块。
