## 二、P0 必须完成项

### 1. Dockerfile + Python ENTRYPOINT

新增提交态 Docker 层：

- `Dockerfile`
- `.dockerignore`
- `docker/gateway_caps.yaml`
- 默认 `ENTRYPOINT`

Dockerfile base image：

```dockerfile
FROM --platform=linux/amd64 python:3.13-slim-bookworm
```

理由：

- 与 `dataagent` dev 环境的 Python 3.13.12 完全对齐，避免 typing / 语法行为漂移。
- 显式 pin Debian Bookworm，确保 base image 在比赛期间不会因 Docker Hub 默认切换而被动改变。
- 比完整 Python 镜像更小，预期镜像总大小 1-1.5 GB。
- 避免 Alpine/musl 对 `numpy`、`pandas` wheels 的兼容风险。

**实施前提（已确认 2026-05-08）**：`dataagent` 环境 Python 版本为 `3.13.12`，base image 选 `python:3.13-slim-bookworm`。详见 §四第 0 步实测结果。

最小可运行 Dockerfile 骨架：

```dockerfile
FROM --platform=linux/amd64 python:3.13-slim-bookworm

WORKDIR /app

COPY pyproject.toml /app/
COPY src/data_agent_langchain /app/src/data_agent_langchain

RUN pip install --no-cache-dir .

COPY docker/gateway_caps.yaml /app/gateway_caps.yaml

ENV PYTHONUNBUFFERED=1 \
    PYTHONDONTWRITEBYTECODE=1

ENTRYPOINT ["python", "-m", "data_agent_langchain.submission"]
```

ENTRYPOINT 调用 `python -m data_agent_langchain.submission`。

日志必须由 Python `logging.FileHandler('/logs/runtime.log')` 控制写入内容。**禁止**在 shell ENTRYPOINT 中使用：

```text
2>&1 | tee /logs/runtime.log
```

原因：`tee` 会把所有 stdout/stderr 原样写入日志，可能包含工具 observation、LLM raw_response 或上下文片段，违反日志脱敏目标。

可选安全增强（不强制）：

```dockerfile
RUN useradd -m -u 1000 dabench && chown -R dabench /app
USER dabench
```

### 2. `gateway_caps.yaml` 固定入镜像

当前 `AgentConfig.action_mode` 默认是 `tool_calling`，runner 会强制读取 `config.observability.gateway_caps_path`。如果镜像内缺少 caps 文件或 `tool_calling=false`，容器启动即失败，所有任务为 0。

v5 固定策略：

- 在仓库新增提交态文件：`docker/gateway_caps.yaml`。
- Dockerfile 显式复制到：`/app/gateway_caps.yaml`。
- 提交入口构造 `AppConfig` 时设置：`gateway_caps_path=Path('/app/gateway_caps.yaml')`。
- 不依赖开发态 `artifacts/gateway_caps.yaml`，避免被 `gateway-smoke` 过程覆盖。

`docker/gateway_caps.yaml` 字面内容：

```yaml
gateway_caps:
  tool_calling: true
  parallel_tool_calls: true
  seed_param: true
  strict_mode: true
```

Dockerfile 复制点：

```dockerfile
COPY docker/gateway_caps.yaml /app/gateway_caps.yaml
```

单测覆盖：

- 文件不存在时给出明确错误。
- 文件存在且 `tool_calling=true` 时可继续。

### 3. 官方 `/output` 扁平化与同目录原子覆盖

内部运行目录可继续保留：

```text
/tmp/dabench-runs/<run_id>/<task_id>/prediction.csv
```

但评测输出必须是：

```text
/output/<task_id>/prediction.csv
```

提交入口要求：

- 每题开始前创建 `/output/<task_id>/prediction.csv` 占位文件。
- 成功后将真实答案覆盖到官方输出路径。
- 不产生 `/output/<run_id>/...`。

占位 CSV 明确为：

```python
PLACEHOLDER_CSV = "result\r\n"
```

使用 `\r\n` 与 `csv.writer` 默认 `lineterminator` 保持一致。

真实答案覆盖必须避免跨挂载点 `os.replace()`。`/tmp/dabench-runs` 与 `/output` 在容器中是不同设备，直接 `os.replace('/tmp/...', '/output/...')` 会触发 `EXDEV: Invalid cross-device link`。

正确流程（成功路径）：

```python
target = Path(f"/output/{task_id}/prediction.csv")
artifact_csv = artifact.prediction_csv_path
if artifact_csv is not None and artifact_csv.exists():
    tmp = target.with_name(".prediction.csv.tmp")
    shutil.copy(artifact_csv, tmp)
    os.replace(tmp, target)
# 否则保留先前写入的占位 CSV，不动 target。
```

要求：

- 临时文件必须与最终 `prediction.csv` 在同一目录下，确保最后一步是同设备原子重命名。
- `TaskRunArtifacts.prediction_csv_path` 在失败 / 超时时为 `None`（`runner.py:329-337`）；此时**不**复制，让占位 CSV 自然保留。
- 临时文件以 `.` 起头，避免被外部工具误识别为 prediction。

### 4. `MODEL_*` 环境变量注入，提交态不读 YAML

提交入口不调用 `load_app_config()`，避免读取本地 YAML 或敏感配置。直接构造核心配置，并显式固定 `graph_mode="plan_solve"`：

```python
dataset = DatasetConfig(root_path=Path("/input"))
run = RunConfig(
    output_dir=Path("/tmp/dabench-runs"),
    max_workers=int(os.environ.get("DABENCH_MAX_WORKERS", 8)),
    task_timeout_seconds=int(os.environ.get("DABENCH_TASK_TIMEOUT_SECONDS", 480)),
)
observability = ObservabilityConfig(
    langsmith_enabled=False,
    gateway_caps_path=Path("/app/gateway_caps.yaml"),
)
agent = AgentConfig(
    model=os.environ.get("MODEL_NAME", "qwen3.5-35b-a3b"),
    api_base=os.environ["MODEL_API_URL"],
    api_key=os.environ.get("MODEL_API_KEY", "EMPTY"),
    action_mode="tool_calling",
)
graph_mode = "plan_solve"
```

约束：

- `MODEL_API_URL` 缺失时立即失败，并只写脱敏错误到 `/logs/runtime.log`。
- `MODEL_API_KEY` 缺失时回退 `EMPTY`。
- `MODEL_NAME` 缺失时回退 `qwen3.5-35b-a3b`。
- 不打印环境变量值，只记录是否存在。
- 只有后续基准验证发现 `plan_solve` 因调用过多导致集中超时时，才考虑切回 `react`。

### 5. per-task 异常隔离与 SIGTERM 处理

正式提交入口不能直接依赖 `run_benchmark()` 的整体 ThreadPool 流程，因为 `future.result()` 重抛异常时可能中断整批任务。

提交态策略：

- 入口先列出 `/input` 下所有 `task_*`。
- 每个任务单独执行 `run_single_task()`。
- 单题 try/except，失败只影响该任务。
- 每题失败写一行 `/logs/failed_tasks.jsonl`。
- 错误摘要只包含：`task_id`、错误类型、错误首行、耗时。
- 不写 observation、raw_response、prompt、上下文片段或密钥。

SIGTERM 处理（必须在主线程注册）：

```python
import signal
import threading

_shutting_down = threading.Event()

def _on_sigterm(signum, frame):
    # handler 只能改标志，不能抛异常
    _shutting_down.set()

# 必须在主线程里注册；ThreadPoolExecutor worker 不会触发 handler
signal.signal(signal.SIGTERM, _on_sigterm)

with ThreadPoolExecutor(max_workers=run.max_workers) as executor:
    futures = []
    for task_id in task_ids:
        if _shutting_down.is_set():
            break
        futures.append(executor.submit(_run_one, task_id))
    executor.shutdown(wait=True, cancel_futures=False)
```

约束：

- `signal.signal()` 必须在**主线程**调用且 handler 只在主线程触发；不能在 worker 线程注册。
- handler 不能抛异常，只能 set `threading.Event` / 全局标志。
- 主线程派发循环每次 submit 前检查 `_shutting_down.is_set()`。
- `executor.shutdown(wait=True, cancel_futures=False)` 等待 in-flight 任务自然结束。
- 依赖 runner 现有 `_reap_process()` 对子进程做 terminate/kill 兜底。

该处理与规则 §4.2 的「已写入结果仍参与评分」语义一致，可尽量保住已完成任务输出。

### 6. `/logs` 脱敏与 summary schema

`/logs` 仅用于排错，不应复制完整 trace。

允许写入：

- `/logs/runtime.log`
- `/logs/failed_tasks.jsonl`
- `/logs/submission_summary.json`

禁止写入：

- `trace.json`
- LLM raw_response
- 工具 observation
- 上下文文件内容片段
- prompt 全文
- API URL / API Key 的实际值
- 失败堆栈全文

`/logs/submission_summary.json` 最小 schema：

```json
{
  "task_count": 60,
  "succeeded_count": 55,
  "failed_count": 5,
  "elapsed_seconds": 7234.2,
  "started_at": "2026-05-08T03:00:00Z",
  "finished_at": "2026-05-08T05:00:34Z",
  "max_workers": 8,
  "task_timeout_seconds": 480,
  "graph_mode": "plan_solve"
}
```

明确不包含：

- model name 与 api_base 的实际值。
- task 级 trace / observation / answer 内容。
- 失败堆栈全文。

### 9. LangSmith 提交态显式关闭

`observability/tracer.py` 已有双重保护：`langsmith_enabled=False` 或缺少 `LANGSMITH_API_KEY` / `LANGCHAIN_API_KEY` 时返回空 callback。v5 仍要求提交入口显式关闭并测试守护：

- `observability.langsmith_enabled=False`
- 启动时清理 `LANGSMITH_*`、`LANGCHAIN_API_KEY`、`LANGCHAIN_TRACING_V2`

```python
for key in list(os.environ):
    if key.startswith("LANGSMITH_") or key in {"LANGCHAIN_API_KEY", "LANGCHAIN_TRACING_V2"}:
        os.environ.pop(key, None)
```

这是合规声明与防回归措施，不是当前代码的直接缺陷。

## 三、P1 建议完善项

### 1. 放宽 `task.json` schema

当前 `_load_task_record()` 要求键集完全等于：

```text
task_id, difficulty, question
```

建议改为：

```python
required = {"task_id", "difficulty", "question"}
missing = required - set(payload)
if missing:
    raise DatasetError(...)
# 多余字段忽略，仅取 required 三键
```

该项提高对官方数据集字段扩展的鲁棒性。

### 2. A/B 榜运行预算分档

| 场景 | max_workers | task_timeout_seconds | 说明 |
| --- | ---: | ---: | --- |
| A 榜约 60 题 | 4 | 600 | 当前默认基本可用 |
| B 榜约 320 题 | 8 | 480 | 理论上限约 5.3h，留冷启动和失败缓冲 |

提交入口设置全局软截止：运行到第 11 小时后停止派发新任务（与 §二.5 SIGTERM 共享 `_shutting_down` 标志）。

### 3. `None` 写 CSV 行为测试

规则会把空值统一处理，但不能把 Python `None` 写成字符串 `"None"`。新增测试确保：

- `answer.rows` 中的 `None` 写盘为空 cell。
- 空 cell 能被标准 CSV reader / pandas 正常解析。

### 4. 密钥全文件泄漏防回归测试

mock 一个明显的 API key 字符串，例如：

```text
sk-MOCK-DO-NOT-LEAK-12345
```

注入 `MODEL_API_KEY` 后运行最小任务，断言以下路径下所有文本文件都不包含该字符串：

- `/logs/runtime.log`
- `/logs/failed_tasks.jsonl`
- `/logs/submission_summary.json`
- `/output/**/prediction.csv`
- 内部 `/tmp/dabench-runs/<run_id>/summary.json`
- 内部 `/tmp/dabench-runs/<run_id>/<task_id>/trace.json`
- 内部 `/tmp/dabench-runs/<run_id>/<task_id>/metrics.json`

`trace.json` 与 `metrics.json` 是 `MetricsCollector` / `build_run_result()` 输出，理论上不应含 api_key。但 LLM client 配置若被任何中间层 `__repr__` / `dict()` 序列化，就可能间接泄漏。该测试是最直接的防回归。

### 5. CLI 过期文案修正

`gateway-smoke` 末尾仍有旧文案（`cli.py:75`），建议同步为当前 `tool_calling` 默认状态。该项为 P2 清理，不阻塞正式提交。

- 每题开始先写 `result\r\n` 占位 CSV。
- 成功结果通过同目录临时文件 + `os.replace()` 原子覆盖占位 CSV。
- 失败任务保留占位 CSV（`prediction_csv_path is None` 分支）。
- 不产生 `/output/<run_id>/...`。

**异常与信号**：

- per-task 失败不阻断其他任务。
- 收到 SIGTERM 后停止派发新任务，并尽量等待 in-flight 写盘。

**脱敏**：

- `/logs/runtime.log`、`failed_tasks.jsonl`、`submission_summary.json`、`/output/**`、内部 `trace.json`、`metrics.json`、`summary.json` 均不含 mock observation / raw_response / API key。
- `None` 写 CSV 为空 cell。

**元测试 / dataset**：

- `pyproject.toml` 主依赖**不含** `datasets` / `duckdb` / `huggingface-hub` / `openpyxl` / `polars` / `pyarrow` / `rich`。
- `pyproject.toml` 删除 `dabench` entry point，保留 `dabench-lc`。
- `task.json` 含多余字段时仍能加载。
- `task.json` 缺三键之一时抛 `DatasetError`。

## 五、正式提交检查清单

- 镜像名符合 `<team_id>:v<N>`。
- 归档名符合 `<team_id>_v<N>.tar.gz`。

- 镜像建议小于 2 GB，必须小于 10 GB。
- 镜像基于 `python:3.13-slim-bookworm` 或经 dev 环境验证兼容的 slim Python 镜像。
- 镜像内不包含 `.env`、本地 API key、`configs/react_baseline.example.yaml` 或其他敏感 YAML。

- 容器不修改 `/input`。
- 输出路径为 `/output/<task_id>/prediction.csv`。
- 成功覆盖使用 `/output/<task_id>/.prediction.csv.tmp` 同目录临时文件。
- 失败任务仍有可解析占位 CSV（`result\r\n`）。
- 每个任务失败不影响其他任务继续运行。
- SIGTERM 后停止派发新任务，等待 in-flight 写盘。
