## 建议修复方向

### 1. 重复失败熔断

同一工具、同一路径、同一错误连续失败 2 到 3 次后，应强制停止该调用路径并 replan。

适用任务：

- `task_350`
- `task_355`
- `task_408`
- `task_418`

### 2. 工具调用前做资产约束

工具调用路径必须来自 `list_context` 返回的真实文件集合。不存在的路径应在调用前被拦截，而不是消耗工具步数。

### 3. SQL 查询绑定真实 schema

SQL 只能引用 `inspect_sqlite_schema` 返回的实际表名。出现 `no such table` 后，应把错误表名加入禁用集合并重写计划。

### 4. 文档 fallback 策略

当目标实体文件不是 CSV 或 DB 时，应自动尝试同名/相关文档：

- `csv/member.csv` 不存在但有 `doc/member.md` 时，改为解析 `doc/member.md`。
- `db/results.db` 不存在但有 `doc/Laboratory.md` / `doc/Patient.md` 时，改为文档抽取。
- 数据库缺少 `races` 表但有 `doc/races.md` 时，改为文档查找 race metadata。

### 5. 部分答案收敛策略

当已得到核心数值但缺少辅助映射时，应允许：

- 提交可验证字段；或
- 提交带缺失说明的结果；或
- 明确停止并报告缺失资产。

这能避免 `task_330` 这种已找到比分但未提交的失败。

### 6. 限制 `execute_python` 访问范围

`task_396` 中 agent 执行了类似 `os.walk('/')` 的系统级扫描。应考虑限制 `execute_python` 的当前工作目录或拦截上下文目录外的文件访问。

## 优先级建议

### P0：防止死循环和无效重复调用

优先实现重复失败熔断、路径存在性约束和 SQL schema 约束。这会直接覆盖 4 个最明确失败任务。

### P1：增强文档 fallback

处理 CSV/DB 缺失但文档存在的情况，覆盖 `task_355`、`task_418`、`task_408` 等混合数据源任务。

### P2：增强语义关联与最终答案收敛

改进 exact match 失败后的语义搜索、文档关联和部分结果提交策略，覆盖 `task_330`、`task_344`、`task_352`、`task_396`。
