## 已修复问题

### RAG-M4-POST-01：图执行前初始化失败时 corpus handles 泄漏

**严重级别：中**

**现象**

`run/runner.py::_run_single_task_core` 先调用 `_build_and_set_corpus_handles`，随后才执行 `_llm_for_action_mode`、编译图、构造 callbacks 和 `compiled.invoke`。旧代码只在 `compiled.invoke` 的 `finally` 中调用 `clear_current_corpus_handles()`。如果 `_llm_for_action_mode` 或其它 invoke 之前的初始化步骤抛错，当前 context 中已经写入的 corpus handles 不会被清理。

**影响**

默认生产路径每个 task 使用子进程，进程退出会自然回收，因此主线风险较低。但单测、`task_timeout_seconds <= 0` 的同步调试路径、或未来复用同进程 runner 的路径会受到影响：后续 task 可能读到上一 task 的 corpus handles，导致召回串味。

**修复**

将 `clear_current_corpus_handles()` 的 `finally` 边界外扩到“RAG handles 构建之后至 invoke 完成”的整个阶段，确保初始化、编译、callback 构造、invoke 任一环节失败都会清理 contextvar。

**修改文件**

- `src/data_agent_langchain/run/runner.py`
- `tests/test_phase10_rag_runner.py`

**新增回归测试**

`tests/test_phase10_rag_runner.py::test_runner_clears_corpus_handles_when_setup_fails_before_invoke`

该测试先 monkeypatch `_build_and_set_corpus_handles` 写入一个 sentinel handles，再让 `_llm_for_action_mode` 在 invoke 前抛错。修复前测试失败，修复后通过。
