      Memory MVP 推荐接入

      1
      工具成功后写结构化知识`tool_node` 在 `read_csv` / `read_json` / `inspect_sqlite_schema` 成功后提取 schema、行数估计、路径信息。
      2
      MemoryWriter 只接受 dataclass`DatasetKnowledgeRecord` 与 `ToolPlaybookRecord` 明确禁止 question、answer、approach、hint。
      3
      Retriever 按 namespace 召回`planner_node` / `model_node` 用 `dataset:{name}` 召回少量结构化事实。
      4
      prompts.py 白名单渲染只把 file path、columns、dtype、row_count 等字段写入 prompt。

      Corpus RAG 推荐接入

      1
      离线 ingest把外部说明文档、领域知识、规则文档切片，写入 corpus store。
      2
      检索与 Memory 解耦Corpus retriever 读取 `corpus:{name}`，不要与 dataset_knowledge 混在一个自由文本 namespace。
      3
      轻量摘要进 RunStatestore / retriever 句柄不进 `RunState`，最多记录 `memory_hits` / recall summary 供 audit。
      4
      按节点注入规划类信息给 `planner_node`，执行上下文给 `model_node`，不要所有节点拿同一个 Facade。

      单任务执行流

      1
      入口构造 AppConfig本地从 `cli.py + load_app_config` 进入，容器从 `submission.py + MODEL_* env` 进入。
      2
      runner 构造初始 RunState字段包含 task_id、question、context_dir、mode、action_mode、max_steps 等。
      3
      选择外层图`react` 使用 ReAct 图；`plan_solve` 先进入 planner，再进入共享 execution 子图。
      4
      执行 T-A-O 循环`model_node → parse_action_node → gate_node → tool_node → advance_node`。
      5
      finalize 落盘写出 trace、metrics、prediction.csv 或 failure_reason。

      RAG 与记忆模块应该怎么加

      核心原则：Memory 是语义复用载体，LangGraph checkpointer 是状态恢复机制，二者不要混用。

      新增模块职责推荐位置接入节点
      MemoryConfig三态开关、store 后端、检索数量`config.py`runner / node 通过 AppConfig 读取
      MemoryRecordKV 信封，带 namespace、kind、payload`memory/base.py`store / retriever / writer
      DatasetKnowledgeRecordfile_path、schema、row_count、dtype`memory/records.py``tool_node` 写，`planner_node` 读
      ToolPlaybookRecord工具参数模板、前置条件、常见失败`memory/records.py``tool_node` 写，`model_node` 读
      JsonlMemoryStoreappend-only KV 存储，低运维成本`memory/stores/jsonl.py`通过 factory 构造
      ExactNamespaceRetriever按 namespace 列出结构化记录`memory/retrievers/exact.py``planner_node` / `model_node`
      Corpus RAG外部文档 ingest、chunk、检索`memory/rag/``planner_node` / `model_node`

      建议的最小实现顺序

      1. 实现 `MemoryStore`、`MemoryWriter`、`ExactNamespaceRetriever`。
      2. 在 `tool_node` 成功工具调用后写 `dataset_knowledge`。
      3. 在 `planner_node` 入口召回 dataset facts。
      4. 在 `prompts.py` 增加白名单渲染函数。
      5. 增加 `memory.mode=disabled` 与 `read_only_dataset` 测试。
      6. 稳定后再新增 `memory/rag/` 做 corpus ingest。

      不要一开始做什么

      • 不要做跨 task episodic memory。
      • 不要把历史答案、解题思路、question、hint 写进长期记忆。
      • 不要把 store / retriever / writer 对象放进 `RunState`。
      • 不要让所有节点拿一个万能 `MemoryFacade`。
      • 不要在图节点循环里构建重型向量索引。

      风险与测试策略

      RAG 与长期记忆会改变模型输入，必须把公平性、可复现性和审计能力设计在第一版里。

      风险：跨 task 答案泄露

      对策：MemoryWriter 只接受结构化 dataclass，字段层面不允许 question、answer、approach、hint、summary。

      风险：破坏 RunState 可序列化

      对策：RunState 只保留轻量 summary，store / retriever / writer 在节点入口通过 factory 构造。

      风险：召回污染 prompt

      对策：`prompts.py` 白名单渲染，只注入允许字段，并记录 memory_recall 事件。

      风险：与 baseline parity 混淆

      对策：保留 `memory.mode=disabled`，任何对比实验都能关闭记忆路径。

      建议测试清单

      • `JsonlMemoryStore.put/list`：验证 KV 写入、namespace 隔离、limit 生效。
      • `ExactNamespaceRetriever.retrieve`：验证按 namespace 返回、空 namespace 返回空列表。
      • `MemoryWriter.write_dataset_knowledge`：验证结构化字段可写入。
      • `read_only_dataset`：允许写 dataset knowledge，拒写 tool playbook。
      • `disabled`：所有写入 no-op，所有 retrieve 返回空。
      • `RunState` picklability：确认没有非 pickle 对象进入 state。
      • `prompts.py` 白名单渲染：payload 中额外字段不会进入 prompt。
      • `memory_recall` event：每次 retrieve 都进入 metrics / trace audit。
