# Benchmark Max-Steps 失败修复计划

本文档给出针对 `20260510T130735Z` 中 `task_344`、`task_352`、`task_396` 三个 `max_steps` 失败任务的分阶段修复计划，目标是在不破坏现有 LangGraph / tool-calling 架构的前提下减少 `no_tool_calls`、重复探索和未提交答案问题。

---

## 0. 审查结论（2026-05-11）

### 0.1 本次审查确认的有效方向

- `sanitize.py` 的 `ERROR_SANITIZED_RESPONSE` 确实是 JSON-action 风格；在 `tool_calling` 模式下经 `memory/working.py` 回放给模型，存在模式冲突。
- `parse_action.py` 在 `tool_calling` 模式下会把空 tool calls 记为 `Model returned no tool_calls.`，且 `advance_node.py` 对 `parse_error` 默认继续，这会消耗 `max_steps`。
- `finalize.py` 的固定失败字符串是 parity 契约，计划中“不修改固定失败字符串”的原则正确。
- `prompts.py` 当前已为 `tool_calling` 去掉一部分 fenced JSON 规则，但仍需要新增回归测试，防止后续 prompt 改动重新引入 JSON-action 约束。
- `model_node.py` 是最合适的 progress hint 接入点，因为它集中构建 ReAct / Plan-and-Solve 执行消息，并能读取 `steps`、`step_index`、`max_steps`、`answer`、`action_mode`。

### 0.2 原计划需要修正的点

- **单任务 CLI 命令写法错误**：`run-task` 的 `task_id` 是位置参数，不是 `--task-id` 选项；正确形式是 `run-task task_396 --config ...`。
- **缺少 `parallel_tool_calls` 合约修复**：当前 `bind_tools_for_gateway()` 会把 `caps.parallel_tool_calls=True` 原样传给 LangChain，但 `parse_action.py` 明确拒绝多 tool calls；应禁止把 `True` 传入绑定参数。
- **现有测试需要同步更新**：`tests/test_phase25_working_memory.py::test_render_step_messages_tool_calling_sanitize_uses_placeholder` 当前断言 tool-calling 也使用 JSON placeholder；实现 Task 1 时必须改成断言使用 tool-calling 专用 placeholder。
- **progress hint 应该 action-mode aware**：`no_tool_calls` 的恢复提示只适用于 `tool_calling`；`json_action` fallback 不应看到 “bound tool call” 指令。
- **近上限提示不要自动 answer**：第一轮只做提示，不生成答案、不改变 `advance_node` 路由，避免把不确定答案强行提交。
- **实现阶段建议使用独立分支 / worktree**：计划审查阶段不创建 worktree；真正改源码时建议先确认 `git status` 干净，并按项目工作流在隔离分支执行。

## 2. 根因假设

### 2.1 直接原因

三个任务都没有调用 `answer`，因此 `finalize_node` 在 `answer is None` 时写入固定失败原因：

```text
Agent did not submit an answer within max_steps.
```

### 2.2 代码路径

相关控制流：

```text
model_node
  -> parse_action_node
  -> gate_node
  -> tool_node 或跳过 tool_node
  -> advance_node
  -> model_node 或 END
  -> finalize_node
```

关键代码：

- `src/data_agent_langchain/agents/parse_action.py`
  - `tool_calling` 模式下空 tool calls 会变成 `__error__`。
  - 错误文本为 `Model returned no tool_calls.`。

- `src/data_agent_langchain/agents/advance_node.py`
  - `parse_error` 不触发 replan。
  - 默认继续下一轮。
  - 到 `step_index >= max_steps` 才结束。

- `src/data_agent_langchain/agents/finalize.py`
  - 没有 answer 时写固定 max_steps failure。

- `src/data_agent_langchain/memory/working.py`
  - `__error__` / `__replan_failed__` 会被替换为 `ERROR_SANITIZED_RESPONSE`。

- `src/data_agent_langchain/agents/sanitize.py`
  - 当前 placeholder 是 JSON-action 风格：

```python
{
    "thought": "My previous response had a format error. I must return a valid JSON with keys: thought, action, action_input.",
    "action": "__error__",
    "action_input": {},
}
```

这与 `tool_calling` 模式冲突：tool-calling 模式要求模型下一轮发起 bound tool call，而不是在 message content 中返回 JSON action。

### 2.3 深层问题

本轮失败包含三个层面：

1. **tool-calling error recovery 不匹配**
   - parse error 后的恢复提示仍在要求 JSON action。
   - 这会加重 `Model returned no tool_calls.` 循环。

2. **无进展循环缺少保护**
   - 连续 `list_context`、连续 `no_tool_calls`、重复相似 Python 探索没有触发强制纠偏。
   - `task_396` 和 `task_344` 体现明显。

3. **接近步数上限时没有提交压力**
   - 当剩余步骤很少时，模型仍继续探索。
   - 应明确提示：如果已有足够证据，优先 `answer`；如果证据不足，也应基于已观察数据给出当前最好表格，而不是继续列目录。

## 4. 推荐方案

### 方案 A：最小稳定性修复，优先推荐

核心改动：

1. 为 `tool_calling` 模式提供专用 error recovery 文本。
2. 明确禁止 `parallel_tool_calls=True` 进入绑定参数，保证模型侧协议和解析器“一次只接受一个 tool call”的约束一致。
3. 在 prompt / scratchpad 中增加剩余步数和提交压力提示。
4. 增加无进展检测，只做软提示，不自动生成答案。
5. 用三个失败 trace 的模式写回归测试。

优点：

- 改动小。
- 风险可控。
- 不改变工具执行协议。
- 不影响 `answer` 工具结构。
- 先修复已确认的协议矛盾，再增加行为提示，便于分阶段回滚。

缺点：

- 不能保证所有 hard 任务都 100% 成功。
- 对模型质量仍有依赖。

### 方案 B：强制 fallback answer

当剩余步数少且没有 answer 时，自动让模型进入“最后提交”模式，甚至在某些条件下由代码构造失败原因或强制提交当前候选结果。

不推荐作为第一步，原因：

- 容易提交错误答案。
- 需要定义“候选结果”的来源和可信度。
- 会增加系统复杂度。

### 方案 C：退回 `json_action`

对困难任务使用 `json_action` fallback。

不推荐作为主方案，原因：

- 当前默认已经切到 `tool_calling`。
- 47/50 成功说明 tool-calling 总体可用。
- 回退会掩盖 tool-calling recovery 的真实问题。

最终建议：**执行方案 A**。

## Task 3：增加无进展检测的纯函数

### 目标

用纯函数识别明显无进展模式，但先只生成提示文本，不自动结束、不自动 answer，降低风险。

### 相关文件

- 新建：`src/data_agent_langchain/agents/progress_guard.py`
- 新建：`tests/test_progress_guard.py`

### 检测规则

第一版只做三个规则，并且恢复提示必须区分 `action_mode`：

1. 最近 8 步中 `list_context` 次数 >= 5。
2. `action_mode == "tool_calling"` 且最近 8 步中 `__error__` 错误包含 `no tool_calls` 次数 >= 3。
3. 剩余步数 <= 8 且 `answer is None`。

```python
from __future__ import annotations

from typing import Any, Sequence

from data_agent_langchain.agents.runtime import StepRecord


def build_progress_hint(
    steps: Sequence[StepRecord],
    *,
    step_index: int,
    max_steps: int,
    answer: Any | None,
    action_mode: str = "tool_calling",
) -> str:
    hints: list[str] = []
    recent = list(steps)[-8:]
    remaining = max(0, int(max_steps) - int(step_index))

    list_context_count = sum(1 for step in recent if step.action == "list_context")
    if list_context_count >= 5:
        hints.append(
            "Progress warning: repeated `list_context` calls were observed. "
            "Do not inspect the directory again unless a new path is needed; move to computation or final answer."
        )

    no_tool_call_count = sum(
        1
        for step in recent
        if step.action == "__error__"
        and "no tool_calls" in str(step.observation.get("error", "")).lower()
    )
    if action_mode == "tool_calling" and no_tool_call_count >= 3:
        hints.append(
            "Recovery warning: recent responses failed with `no tool_calls`. "
            "The next response must use exactly one bound tool call."
        )

    if answer is None and remaining <= 8:
        hints.append(
            f"Step budget warning: only {remaining} step(s) remain. "
            "If the result is known, call the `answer` tool now. "
            "Do not repeat `list_context`. "
            "If critical evidence is still missing, make at most one targeted read or compute call."
        )

    return "\n".join(hints)
```

## 7. 风险与回滚

### 7.1 风险

- tool-calling sanitize 文本改变可能影响模型行为。
- progress hint 可能让模型过早提交答案。
- near-limit hint 可能降低验证充分性。

### 7.2 降低风险的方法

- 第一版只做 prompt/hint，不做自动 answer。
- 所有逻辑使用纯函数，便于测试。
- 不改 `answer` 工具和 `finalize` 语义。
- 保持 `json_action` 模式原有 sanitize 文本。

### 7.3 回滚方式

如果完整 benchmark 成功率下降：

1. 先禁用 `progress_hint` 接入，只保留 tool-calling sanitize 修复。
2. 如果仍下降，再回滚 `tool_calling` sanitize 文本。
3. 保留新增测试，但根据最终设计调整断言。
