## P1：deterministic execution loop 与 answer forcing

P1 分成两个互不混合的主题。

### 5.1 execute_python deterministic loop

代表任务：

```text
task_199
task_418
```

现象：

- `task_199` 在 variant 中结果生成失败，`execute_python` 中 `name 'frm' is not defined` 重复 `39` 次。
- `task_418` stale DB path 已消除，但 `execute_python` 调用 `51` 次，且多次成功但没有推进到答案。

初步方向：

- 对相同 Python exception signature 做 deterministic failure loop guard。
- 对成功但输出高度重复、无新信息的 Python 调用做 success-no-new-info guard。
- 触发后应要求 replan、改代码、转向已知 observation 或提交答案。

非目标：

- 不在 SQL/schema guard 轮次中处理。
- 不把 Python loop 和 SQL loop 共用同一个 signature 规则。

建议进入条件：

- P0 schema loop guard 最小验收完成。
- 已保存 `task_199` / `task_418` 的 trace 统计和重复 signature。

### 5.2 repeated successful tool call / answer forcing

代表任务：

```text
task_283
```

现象：

- baseline score `1.0`。
- variant 结果生成失败。
- trace 显示 `execute_context_sql` 成功调用 `57` 次，但没有提交答案。

初步方向：

- 检测连续成功工具调用是否返回稳定结果。
- 如果结果已经足够回答，提示或强制转入 answer synthesis。
- 如果连续成功结果无新增信息，触发 replan 或 answer forcing。

非目标：

- 不与 SQL/schema mismatch guard 混合。
- 不在没有足够 observation 的情况下强制生成答案。

建议进入条件：

- 完成 P0。
- 明确 answer forcing 的安全边界。

### 5.3 task_173 answer regression diff review

代表任务：

```text
task_173
```

现象：

- baseline score `1.0`。
- variant score `0.0`。
- 仍有 prediction，但答案不一致。

初步方向：

- 对比 baseline / variant prediction.csv。
- 对比 trace 中关键工具输出和最终 answer。
- 判断是否由 RAG/memory prompt 变化、随机性、工具使用差异或评分边界造成。

非目标：

- 不在没有 diff 证据前归因到 stale path guard。

建议进入条件：

- P0 后或 P1 前均可单独做。
- 如果 diff 显示是系统性 regression，应提升优先级。

## P2：list/search/tool-selection/data-understanding 分类修复

### 6.1 list_context / search loop

代表任务：

```text
task_352
task_396
```

现象：

- `task_352`：大量 `list_context` / `read_doc` loop，无 missing path / schema error。
- `task_396`：大量 `list_context` loop，无 missing path / schema error。

初步方向：

- 检测重复 `list_context` 且 context 文件集无变化的循环。
- 如果已读过相关 doc/json 且无新增文件，要求转入内容分析或 answer synthesis。
- 区分“需要重新发现文件”和“已经发现但不行动”。

非目标：

- 不把所有重复 `list_context` 都视为错误。
- 不影响第一轮 discovery gate。

### 6.2 CSV-as-DB tool selection misuse

代表任务：

```text
task_330
```

现象：

- agent 对 `csv/Match.csv` 调用 `execute_context_sql`。
- 出现 `file is not a database` 多次。
- 伴随 `csv/Team.csv` missing path 与 `list_context` loop。

初步方向：

- 工具层或 pre-tool guard 阻止对 `.csv` 使用 `execute_context_sql`。
- 错误反馈明确建议改用 `read_csv` 或 `execute_python` + pandas。
- 如果任务需要不存在的 mapping 文件，应转向当前 context 中已有字段或 docs。

非目标：

- 不在 P0 SQL/schema guard 中处理，因为这是 tool selection / file type mismatch，不是 schema mismatch。

### 6.3 data / document understanding

代表任务：

```text
task_344
task_352
```

现象：

- `task_344`：数据存在，但模型未正确抽取 / 计算 WBC 与 fibrinogen 条件。
- `task_352`：文档和 CSV 存在，但未稳定完成 Yearly Kickoff / October Meeting 对比。

初步方向：

- 增强任务内 evidence extraction 流程。
- 针对文档精确匹配任务，要求先定位原文 span，再执行计算。
- 针对 CSV + doc 联合任务，要求明确 join / filter 条件。

非目标：

- 不优先做 prompt 大改。
- 不在 deterministic loop guard 完成前调整 RAG top-k 或 dynamic recall。

### 7.2 task-scoped dataset memory

背景：

- 第一轮通过 `memory.inject_dataset_facts=False` 避免了裸 `dataset:input` prompt 注入污染。
- 长期如果需要恢复 dataset memory 收益，应设计 task-scoped dataset memory。

初步方向：

- 不恢复裸 `dataset:input` 自动 prompt 注入。
- 将 dataset facts 与 task identity / context fingerprint 绑定。
- 旧 namespace 仅用于离线分析或 curated shared memory 候选。

进入条件：

- P0 / P1 完成。
- 明确 dataset memory 的收益样本，不只为了恢复旧行为。

### 7.3 curated shared memory

背景：

- 跨任务 memory 有潜在收益，但未经筛选的 dataset facts 会污染路径和 schema。

初步方向：

- 只保存稳定、通用、非路径依赖的 shared facts。
- 明确来源、适用范围和失效条件。
- 不允许 shared memory 直接提供当前任务工具路径。

进入条件：

- task-scoped dataset memory 设计清楚后再做。

### 7.4 RAG retrieval 参数再评估

候选项：

- `memory.rag.retrieval_k`
- `memory.rag.prompt_budget_chars`
- dynamic recall
- planning-before-recall

暂缓原因：

- 当前主要瓶颈不是 recall 数量，而是 memory/tool authority 与 deterministic loop。
- 在 loop guard 未完成前调参会混淆归因。

进入条件：

- P0 / P1 完成。
- R6 E3 证明召回质量或预算确实是瓶颈。
