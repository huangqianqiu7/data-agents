## 二、v3 必须修的实质问题

### 1. **§二.8 依赖瘦身是加法不是减法（P0 级错误）**

v3 提出新增 `[project.optional-dependencies] submission = [...]` 并用 `pip install ".[submission]"`。但这是**加法**：仍然会先安装 `pyproject.toml` 主 `dependencies` 列表里的全部包。

实际仓库的 `pyproject.toml:11-25` 主 `dependencies` 含：

```text
datasets>=3.3.2
duckdb>=1.5.0
huggingface-hub>=0.29.3
openpyxl>=3.1.5
polars>=1.39.0
pyarrow>=23.0.1
rich>=13.9.4
```

经 grep 验证，`src/data_agent_langchain` 源码内**没有**任何 `import datasets` / `import huggingface` / `import polars` / `import pyarrow` / `import duckdb` / `import openpyxl` / `import rich`。这些是 legacy `data_agent_baseline` 的依赖，会让镜像凭空增加数百 MB（仅 `pyarrow` + `polars` 就接近 200 MB）。

**正确策略（减法）**：

- 修改主 `dependencies`，缩成 `data_agent_langchain` 实际需要的最小集：

  ```toml
  dependencies = [
      "langchain-core",
      "langchain-openai",
      "langgraph",
      "openai>=2.28.0",
      "pandas>=2.3.3",
      "pydantic>=2.10.6",
      "pyyaml>=6.0.2",
      "typer>=0.15.2",
      "numpy>=2.2.6",
  ]
  ```

- 把 baseline 专属依赖移到 `[project.optional-dependencies]`：

  ```toml
  [project.optional-dependencies]
  baseline = [
      "datasets>=3.3.2",
      "duckdb>=1.5.0",
      "huggingface-hub>=0.29.3",
      "openpyxl>=3.1.5",
      "polars>=1.39.0",
      "pyarrow>=23.0.1",
      "rich>=13.9.4",
  ]
  dev = [
      "pytest>=8.3.5",
      "ruff>=0.11.0",
  ]
  ```

- Dockerfile 用：

  ```text
  pip install --no-cache-dir .
  ```

  不带任何 extras。

**注意还需同步处理**：`pyproject.toml:34` 的 entry point `dabench = "data_agent_baseline.cli:main"` 引用的模块不存在（`src/data_agent_baseline/` 已归档到 `src/废弃/data_agent_baseline/`，hatchling 默认包发现不会把它当顶层包）。安装本身不会失败，但 `dabench` CLI 调用会 `ImportError`。**建议在 v3 §二.8 同步删除该 entry point**，只保留 `dabench-lc`。

### 2. **§二.3 `os.replace()` 跨挂载点会 EXDEV 失败**

v3 写「先写临时文件，再 `os.replace()`」是对的，但**没明确临时文件位置**。

在容器里 `/tmp/dabench-runs/...` 与 `/output/<task_id>/` 是**不同 mount**（`/output` 是 docker volume，`/tmp` 通常是容器内 overlayfs 或 tmpfs）。Linux 的 `os.replace()` 在跨设备时抛 `OSError: [Errno 18] Invalid cross-device link`。

**正确写法**：临时文件必须与最终路径**同目录**，例如：

```python
target = Path(f"/output/{task_id}/prediction.csv")
tmp = target.with_name(".prediction.csv.tmp")
tmp.write_bytes(real_csv_bytes)
os.replace(tmp, target)  # 同目录原子重命名
```

如果 runner 内部的 `prediction.csv` 必须从 `/tmp/dabench-runs/...` 搬过去，应：

```python
import shutil
shutil.copy(internal_csv, tmp)  # 跨设备拷贝
os.replace(tmp, target)         # 同设备原子重命名
```

建议在 v3 §二.3 加一行明确这一约束。

## 三、v3 还应补充的项

### 3. **§二.2 `gateway_caps.yaml` 内容来源未指明**

v3 说「在镜像中放置 `/app/gateway_caps.yaml`」「文件内容只描述官方 OpenAI 风格 Tool Calling 支持能力」，但没给字面值，也没说从哪里来。开发实施时要么从仓库 `artifacts/gateway_caps.yaml` 复制，要么手写。

建议在 §二.2 加一段直接给字面值（与当前 `artifacts/gateway_caps.yaml` 一致，且与 rules §5.3 vLLM 部署能力一致）：

```yaml
gateway_caps:
  tool_calling: true
  parallel_tool_calls: true
  seed_param: true
  strict_mode: true
```

并在 Dockerfile 中：

```dockerfile
COPY docker/gateway_caps.yaml /app/gateway_caps.yaml
```

放在仓库内独立路径（如 `docker/gateway_caps.yaml`），与开发态 `artifacts/gateway_caps.yaml` 解耦，避免开发跑 `gateway-smoke` 误覆盖提交态文件。

### 4. **§四.3 Dockerfile base image 未指明**

v3 §四.3 只说「支持 linux/amd64」。没说 base image。建议明确：

```dockerfile
FROM --platform=linux/amd64 python:3.10-slim
```

理由：

- `python:3.10-slim` 体积约 200 MB，镜像总大小可控制在 1-1.5 GB。
- 与仓库 `pyproject.toml` 的 `requires-python = ">=3.10"` 兼容。
- 避免 alpine 在 numpy / pandas 编译时的 musl 兼容性问题。

如果开发环境 Python 版本是 3.11 或 3.12，应改用对应 slim 镜像（建议先在 dev 环境用 `python --version` 确认）。

### 5. **SIGTERM 处理（接 §三.2 11h 软截止）**

v3 §三.2 提了「11 小时软截止停止派发新任务」是好的，但规则 §4.2 的 12h 硬上限会先发 SIGTERM 再 30s 后 SIGKILL。提交入口主进程当前没有 SIGTERM handler，默认会立刻死，**正在 in-flight 的任务子进程的部分输出可能丢失**。

建议在 §二.5 或新增 §二.10 加：

- 主进程注册 `signal.SIGTERM` handler，set `_shutting_down=True` 标志。
- 任务派发循环每次取 future 前检查 `_shutting_down`，true 则停止派发。
- ThreadPoolExecutor 用 `shutdown(wait=True, cancel_futures=False)` 等 in-flight 任务自然结束。
- 子进程内的 `_run_single_task_with_timeout` 已有 `_reap_process` 兜底，30s SIGKILL 不会卡住。

实际收益有限（多保住 1-2 题）但符合 §4.2 的"已写入仍参与评分"语义。

### 6. **§四.2 graph_mode 默认值未指明**

`run_single_task()` 函数签名 `graph_mode: GraphMode = "plan_solve"` 是 runner 默认。提交入口应显式选定，避免依赖默认值。建议在 §二 或 §四.2 加一行：

> 提交入口固定 `graph_mode="plan_solve"`，因为它支持 replan 与外层 plan 节点，对 hard / extra_hard 题目更鲁棒。当且仅当后续基准评测发现 plan_solve 因 LLM 调用过多而集中超时时，才考虑切回 `react`。

### 7. **§四.2 `/logs/submission_summary.json` 字段未规定**

v3 §四.2 步骤 10 列出该文件但没列字段。建议在 §二.6 或 §四.2 给最小 schema：

```json
{
  "task_count": 60,
  "succeeded_count": 55,
  "failed_count": 5,
  "elapsed_seconds": 7234.2,
  "started_at": "2026-05-08T03:00:00Z",
  "finished_at": "2026-05-08T05:00:34Z",
  "max_workers": 8,
  "task_timeout_seconds": 480,
  "graph_mode": "plan_solve"
}
```

明确**不**包含：

- model name 与 api_base 的实际值。
- 任何 task 级别的 trace / observation / answer 内容。
- 失败任务的错误堆栈全文（只放在 `failed_tasks.jsonl`）。

### 8. **§四.1 测试列表缺一条 api_key 不进 logs / summary 的断言**

v3 §四.1 测试列表有「`/logs/runtime.log` 不包含 mock observation / raw_response / API key」，但没显式覆盖 `summary.json` / `failed_tasks.jsonl` / `metrics.json` 中也不能含 api_key。

建议测试改为：

> mock 一个明显的 api_key 字符串（如 `"sk-MOCK-DO-NOT-LEAK-12345"`）注入 `MODEL_API_KEY`，运行一个最小任务后断言 `/logs/` 下所有文件、`/output/` 下所有文件、内部 `/tmp/dabench-runs/` 下的 `summary.json` 与 `metrics.json` 全部不含该字符串子串。

这是最直接的密钥泄漏防回归测试。

## 四、备注（不阻塞，可选）

### 9. multiprocessing start method

Linux 默认 `fork`。当前 runner 用默认 context。理论上 fork-after-thread 在 LangGraph + LangChain 异步状态下可能造成偶发 deadlock，但 v2 评测套件 162 个测试全过，实践上 OK。如未来出现间歇性挂死，可考虑 `multiprocessing.set_start_method("spawn", force=True)`，代价是每题 fork 多 0.5-1s。

### 10. gateway_caps 每题重复加载

`runner._llm_for_action_mode` 在每个任务进入时都重新读 `gateway_caps.yaml`（`runner.py:195`）。320 题 × 一次磁盘读 ≈ 几十 ms，可忽略。但如果担心，可在提交入口预加载并缓存到 module-level。属 P3 优化。

### 11. `pyproject.toml` `dabench` entry point 修复

如 §二.1 所述，建议同步删除 `dabench = "data_agent_baseline.cli:main"`，避免误用时 ImportError 把诊断时间浪费在 entry point 而不是真正的提交错误。
