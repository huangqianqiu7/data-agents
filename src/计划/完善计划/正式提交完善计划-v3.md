# 正式提交完善计划 v3

本文档基于 v2 计划、v2 审查结论、`rules.zh.md` 与 `src/data_agent_langchain` 当前代码核对结果，形成正式开工前的最终完善计划。

## 一、当前代码核对结论

`data_agent_langchain` 已具备 LangGraph 后端、任务加载、子进程隔离、超时控制、CSV 写出、Tool Calling 绑定和基础可观测能力。但正式 KDD Cup Docker 提交仍需要补齐提交态入口、容器层、输出布局、异常隔离、日志脱敏和依赖瘦身。

| 事项 | 当前状态 | v3 判断 |
| --- | --- | --- |
| Dockerfile / `.dockerignore` | 仓库根未提供提交态容器文件 | P0 |
| 输出路径 | runner 写到 `<run_id>/<task_id>/prediction.csv` | P0，需扁平化到 `/output/<task_id>/prediction.csv` |
| 模型配置 | `build_chat_model()` 使用 `config.agent.model/api_base/api_key` | P0，提交入口直接从 `MODEL_*` 构造 `AppConfig` |
| Tool Calling caps | `runner.py` 在 `tool_calling` 默认模式下强校验 `gateway_caps.yaml` | P0，缺文件会全批启动失败 |
| 批量异常 | `future.result()` 会重抛异常 | P0，提交态需 per-task try/except |
| 失败 CSV | `answer` 非 dict 时不写 `prediction.csv` | P0，需占位 CSV 兜底 |
| 日志 | trace 可能包含 raw_response / observation | P0，`/logs` 只允许脱敏摘要 |
| LangSmith | `tracer.py` 已有双重保护，默认 false | P0 测试守护，提交入口显式锁死 false |
| `task.json` schema | 当前要求键集完全等于三键 | P1，需容忍官方新增字段 |
| 依赖 | `pyproject.toml` 含 legacy baseline 依赖，未列 LangChain / LangGraph | P0/P1，提交镜像需精简并固化运行时依赖 |

## 二、P0 必须完成项

### 1. Dockerfile + Python ENTRYPOINT

新增提交态 Docker 层：

- `Dockerfile`
- `.dockerignore`
- 默认 `ENTRYPOINT`
- 镜像内固定 `gateway_caps.yaml`

ENTRYPOINT 必须调用：

```text
python -m data_agent_langchain.submission
```

日志必须由 Python `logging.FileHandler('/logs/runtime.log')` 控制写入内容。禁止在 shell ENTRYPOINT 中使用：

```text
2>&1 | tee /logs/runtime.log
```

原因：`tee` 会把所有 stdout/stderr 原样写入日志，可能包含工具 observation、LLM raw_response 或上下文片段，违反日志脱敏目标。

### 2. `gateway_caps.yaml` 升为 P0 并固定镜像路径

当前 `AgentConfig.action_mode` 默认是 `tool_calling`，runner 会强制读取 `config.observability.gateway_caps_path`。如果镜像内缺少 caps 文件或 `tool_calling=false`，容器启动即失败，所有任务为 0。

v3 固定策略：

- 在镜像中放置 `/app/gateway_caps.yaml`。
- 提交入口构造 `AppConfig` 时设置 `gateway_caps_path=Path('/app/gateway_caps.yaml')`。
- 文件内容只描述官方 OpenAI 风格 Tool Calling 支持能力，不含 URL、Key 或私有配置。
- 单测覆盖：文件不存在时给出明确错误；文件存在且 `tool_calling=true` 时可继续。

### 3. 官方 `/output` 扁平化与原子写入

内部运行目录可继续保留：

```text
/tmp/dabench-runs/<run_id>/<task_id>/prediction.csv
```

但评测输出必须是：

```text
/output/<task_id>/prediction.csv
```

提交入口要求：

- 每题开始前创建 `/output/<task_id>/prediction.csv` 占位文件。
- 成功后将真实答案原子覆盖到官方输出路径。
- 覆盖方式为先写临时文件，再 `os.replace()`，避免半截 CSV。
- 不产生 `/output/<run_id>/...`。

占位 CSV 明确为：

```python
PLACEHOLDER_CSV = "result\n"
```

它不是伪造答案，只是保证失败/超时任务仍有可解析 CSV 文件。

### 4. `MODEL_*` 环境变量注入，提交态不读 YAML

提交入口不调用 `load_app_config()`，避免读取本地 YAML 或敏感配置。直接构造核心配置：

```python
dataset = DatasetConfig(root_path=Path("/input"))
run = RunConfig(
    output_dir=Path("/tmp/dabench-runs"),
    max_workers=int(os.environ.get("DABENCH_MAX_WORKERS", 8)),
    task_timeout_seconds=int(os.environ.get("DABENCH_TASK_TIMEOUT_SECONDS", 480)),
)
observability = ObservabilityConfig(
    langsmith_enabled=False,
    gateway_caps_path=Path("/app/gateway_caps.yaml"),
)
agent = AgentConfig(
    model=os.environ.get("MODEL_NAME", "qwen3.5-35b-a3b"),
    api_base=os.environ["MODEL_API_URL"],
    api_key=os.environ.get("MODEL_API_KEY", "EMPTY"),
    action_mode="tool_calling",
)
```

约束：

- `MODEL_API_URL` 缺失时立即失败，并只写脱敏错误到 `/logs/runtime.log`。
- `MODEL_API_KEY` 缺失时回退 `EMPTY`。
- `MODEL_NAME` 缺失时回退 `qwen3.5-35b-a3b`。
- 不打印环境变量值，只记录是否存在。

### 5. per-task 异常隔离

正式提交入口不能直接依赖 `run_benchmark()` 的整体 ThreadPool 流程，因为 `future.result()` 重抛异常时可能中断整批任务。

提交态策略：

- 入口先列出 `/input` 下所有 `task_*`。
- 每个任务单独执行 `run_single_task()`。
- 单题 try/except，失败只影响该任务。
- 每题失败写一行 `/logs/failed_tasks.jsonl`。
- 错误摘要只包含：`task_id`、错误类型、错误首行、耗时。
- 不写 observation、raw_response、prompt、上下文片段或密钥。

### 6. `/logs` 脱敏与最小化

`/logs` 仅用于排错，不应复制完整 trace。

允许写入：

- `/logs/runtime.log`
- `/logs/failed_tasks.jsonl`
- `/logs/submission_summary.json`

禁止写入：

- `trace.json`
- LLM raw_response
- 工具 observation
- 上下文文件内容片段
- prompt 全文
- API URL / API Key 的实际值

测试必须包含：mock observation 字符串后，断言 `/logs/runtime.log` 与 `/logs/failed_tasks.jsonl` 不出现该字符串。

### 7. `.dockerignore` 完整排除清单

`.dockerignore` 至少包含：

```text
.env
.git/
.vscode/
configs/
data/
old/
Comparison/
tests/
test_api.py
test_bench_comparison.py
evaluation_report.csv
src/test_react_parsers.py
src/废弃/
src/LangChain/
artifacts/runs/
artifacts/dabench-lc/
.pytest_cache/
__MACOSX/
**/__pycache__/
**/.ipynb_checkpoints/
*.tar.gz
*.tar
*.bak
*.swp
*.tmp
```

注意：不能排除提交态需要的 `/app/gateway_caps.yaml`。建议不要依赖 `artifacts/` 中的开发态文件，而是在 Docker 构建阶段显式复制一个提交态 caps 文件到 `/app/gateway_caps.yaml`。

构建后检查：

```text
docker build --no-cache --progress=plain --platform=linux/amd64 -t <team_id>:v<N> .
```

目标镜像建议小于 2 GB，显著低于规则 10 GB 上限。如果某个 `COPY` 层异常大，回查 `.dockerignore` 漏项。

### 8. 精简提交态依赖

当前 `pyproject.toml` 主依赖包含 legacy baseline 相关包，例如 `datasets`、`huggingface-hub`、`polars`、`pyarrow` 等，可能显著增大镜像。`data_agent_langchain` 源码还需要显式列出 LangChain / LangGraph 运行时依赖。

建议新增 optional dependency 组：

```toml
[project.optional-dependencies]
submission = [
    "langchain-core",
    "langchain-openai",
    "langgraph",
    "openai",
    "pandas",
    "pyyaml",
    "typer",
    "pydantic",
    "numpy",
]
```

Dockerfile 使用：

```text
pip install ".[submission]"
```

版本应以当前 `dataagent` 环境中实际验证通过的版本为准；若选择使用 `uv.lock`，则 Dockerfile 必须明确采用锁文件安装策略，避免与 `pyproject.toml` 解析方式混用。

### 9. LangSmith 提交态显式关闭

`observability/tracer.py` 已有双重保护：`langsmith_enabled=False` 或缺少 `LANGSMITH_API_KEY` / `LANGCHAIN_API_KEY` 时返回空 callback。v3 中仍要求提交入口显式关闭并测试守护：

- `observability.langsmith_enabled=False`
- 清理 `LANGSMITH_*`、`LANGCHAIN_API_KEY`、`LANGCHAIN_TRACING_V2`

这是合规声明与防回归措施，不是当前代码的直接缺陷。

## 三、P1 建议完善项

### 1. 放宽 `task.json` schema

当前 `_load_task_record()` 要求键集完全等于：

```text
task_id, difficulty, question
```

建议改为：

- 必须包含三键。
- 多余字段忽略。
- 缺少必需字段才抛 `DatasetError`。

该项提高对官方数据集字段扩展的鲁棒性。

### 2. A/B 榜运行预算分档

保留两档即可，删除 v2 中触发条件不明确的“保守低资源”档。

| 场景 | max_workers | task_timeout_seconds | 说明 |
| --- | ---: | ---: | --- |
| A 榜约 60 题 | 4 | 600 | 当前默认基本可用 |
| B 榜约 320 题 | 8 | 480 | 理论上限约 5.3h，留冷启动和失败缓冲 |

提交入口还应设置全局软截止：运行到第 11 小时后停止派发新任务，等待 in-flight 任务收尾并确保已完成结果落盘。

### 3. `None` 写 CSV 行为测试

规则会把空值统一处理，但不能把 Python `None` 写成字符串 `"None"`。新增测试确保：

- `answer.rows` 中的 `None` 写盘为空 cell。
- 空 cell 能被标准 CSV reader / pandas 正常解析。

### 4. CLI 过期文案修正

`gateway-smoke` 末尾仍有旧文案，建议同步为当前 `tool_calling` 默认状态。该项为 P2 清理，不阻塞正式提交。

## 四、实施顺序

### 第 1 步：测试先行

新增或扩展测试，至少覆盖：

- 缺少 `MODEL_API_URL` 时失败并给出脱敏错误。
- `MODEL_API_KEY` 缺失时回退 `EMPTY`。
- `MODEL_NAME` 缺失时回退 `qwen3.5-35b-a3b`。
- `AppConfig.dataset.root_path == /input`。
- `AppConfig.run.output_dir == /tmp/dabench-runs`。
- `AppConfig.observability.gateway_caps_path == /app/gateway_caps.yaml`。
- `AppConfig.observability.langsmith_enabled is False`。
- gateway caps 缺失时明确失败。
- 每题开始先写 `result\n` 占位 CSV。
- 成功结果原子覆盖占位 CSV。
- 不产生 `/output/<run_id>/...`。
- per-task 失败不阻断其他任务。
- `/logs/runtime.log` 不包含 mock observation / raw_response / API key。
- `None` 写 CSV 为空 cell。

### 第 2 步：新增 `data_agent_langchain.submission`

职责：

1. 初始化脱敏 logging。
2. 校验 `/input`、`/output`、`/logs`。
3. 清理 LangSmith 相关环境变量。
4. 从 `MODEL_*` 与 `DABENCH_*` 构造 `AppConfig`。
5. 校验 `/app/gateway_caps.yaml`。
6. 枚举任务并 per-task 执行。
7. 每题先写占位 CSV。
8. 成功后原子覆盖 `/output/<task_id>/prediction.csv`。
9. 失败写 `/logs/failed_tasks.jsonl`。
10. 结束写 `/logs/submission_summary.json`。

### 第 3 步：补 Docker 提交层

新增：

- `Dockerfile`
- `.dockerignore`
- 提交态 `/app/gateway_caps.yaml` 来源

Dockerfile 要求：

- 支持 `linux/amd64`。
- 安装精简提交态依赖。
- 设置 `ENTRYPOINT ["python", "-m", "data_agent_langchain.submission"]`。
- 不复制 `.env`、`configs/`、`data/`、测试目录、废弃源码、迁移文档或历史 artifacts。

### 第 4 步：增强数据集 schema 鲁棒性

修改 `benchmark/dataset.py`：

- 必含三键。
- 多余字段忽略。
- 保持缺字段时报错。

### 第 5 步：本地与容器验证

本地测试：

```text
C:\Users\18155\anaconda3\envs\dataagent\python.exe -m pytest tests/ -q
```

CLI smoke：

```text
C:\Users\18155\anaconda3\envs\dataagent\python.exe -m data_agent_langchain.cli --help
```

容器 smoke 使用最小 fake 任务目录，不直接挂大规模 `data/`：

```text
docker build --platform=linux/amd64 -t <team_id>:v<N> .

docker run --rm \
  -v <sample_tasks>:/input:ro \
  -v <local_output>:/output:rw \
  -v <local_logs>:/logs:rw \
  -e MODEL_API_URL=<模型服务地址> \
  -e MODEL_API_KEY=<密钥或 EMPTY> \
  -e MODEL_NAME=qwen3.5-35b-a3b \
  <team_id>:v<N>
```

如尚无最小任务目录，建议新建 `tests/fixtures/sample_tasks/`，仅放 1-2 个极小任务用于容器 smoke，避免真实大数据和长时间 LLM 调用。

## 五、正式提交检查清单

- 镜像名符合 `<team_id>:v<N>`。
- 归档名符合 `<team_id>_v<N>.tar.gz`。
- 使用 `docker save` 导出，不使用 `docker export`。
- 镜像支持 `linux/amd64`。
- 镜像建议小于 2 GB，必须小于 10 GB。
- 镜像内不包含 `.env`、本地 API key、`configs/react_baseline.example.yaml` 或其他敏感 YAML。
- 镜像内包含 `/app/gateway_caps.yaml`，且 `tool_calling=true`。
- 代码不硬编码真实 `MODEL_API_URL`、`MODEL_API_KEY`、`MODEL_NAME`。
- 容器只访问 `MODEL_API_URL` 指定的内部模型服务。
- 容器不修改 `/input`。
- 输出路径为 `/output/<task_id>/prediction.csv`。
- 每个任务失败不影响其他任务继续运行。
- 失败任务仍有可解析占位 CSV。
- `/logs` 只包含脱敏日志、失败摘要和 summary。
- `runtime.log` 不包含 observation、raw_response、prompt、上下文片段或密钥。
- `langsmith_enabled` 在提交态强制为 false。
- 容器无需交互输入。
- Google Drive 链接权限为 Anyone with the link / Viewer。
- 提交邮件从队长注册邮箱发送。

## 六、v3 优先级表

| 优先级 | 事项 | 说明 |
| --- | --- | --- |
| P0 | Dockerfile + Python ENTRYPOINT | 无容器入口无法提交 |

## 七、最终建议

v3 的实施方向仍是“新增最小提交适配层 `data_agent_langchain.submission`，不大改现有 LangGraph runner”。但与 v2 相比，v3 明确把 `gateway_caps.yaml`、Python logging、占位 CSV 字节内容、`.dockerignore` 补漏和精简提交态依赖提升为正式开工前必须解决的问题。完成 P0 项后再编写 Dockerfile 与容器 smoke，正式提交风险最低。
