## 二、必须补进计划的关键遗漏

### 1. per-task 异常隔离（P0，最高风险）

规则 §4.2 明确：未写出 `prediction.csv` 即 0 分；已写出的仍计分。当前实现存在让单题异常波及整批的链路：

- `DABenchPublicDataset.get_task` 在 `context/` 不存在时抛 `TaskNotFoundError`（`benchmark/dataset.py:85-86`）。
- `_load_task_record` 严格匹配 `{task_id, difficulty, question}` 三键，多 1 个字段就抛 `DatasetError`（`benchmark/dataset.py:36-41`）。

`run_benchmark` 的 ThreadPool 路径里 `future.result()` 会重抛异常（`run/runner.py:438`），未做 try/except 包裹。一题 schema 偏差就会让整批 320 题中断。

**修复方向**：提交入口对每个 task 单独 try/except，失败任务记一行到 `/logs/failed_tasks.jsonl`，继续下一题。

### 2. `task.json` 严格 schema 校验隐患（P1）

`_load_task_record` 要求键集**完全等于** `{task_id, difficulty, question}`。若官方在后续版本里加入 `language` / `version` / `tags` 等任一字段，全数据集会立即崩溃。建议提交前把它放宽为「必含三键即通过、多余键忽略」。该项不属于 Docker 化但不修复就有归零风险。

### 3. 失败任务无 `prediction.csv` 兜底（P0）

`run/runner.py:329-337` 在 `answer` 不是 `dict` 时跳过 CSV 写出。子进程超时或异常的题，目录里只有 `trace.json` 没有 `prediction.csv`，与「文件不存在」等价（直接 0 分）。

**建议**：每题入口先写一个空表头占位 CSV，再在成功时覆盖；这样即便 LangGraph 子进程被 SIGKILL，仍保留可解析 CSV，避免评测器在解析阶段就跳过整题。

### 4. `/logs` 内容审计（P0）

当前 `trace.json` 完整保留 LLM `raw_response` 与工具 observation。后者会回显从 `context/` 读到的任务真实数据片段（CSV 行、JSON 内容、`knowledge.md` 片段）。规则 §3.7 警告日志包含「测试集数据」会取消调试支持资格、严重者取消参赛资格。

**修复方向**：

- `/logs/runtime.log` 只保留 `task_id` + 状态 + 错误堆栈 + 步数 + 耗时。
- `trace.json` 一律留在内部 run 目录，不复制到 `/logs`。
- 失败摘要（`/logs/failed_tasks.jsonl`）只含 `task_id` + 错误类型 + 错误首行，不复制 LLM 输出与上下文片段。

### 5. `.env` 与 `configs/react_baseline.example.yaml` 必须显式 dockerignore（P0）

仓库根有 `.env`（已被 gitignore），以及 `configs/react_baseline.example.yaml`（已知含 API key）。`.dockerignore` 至少应包含：

```text
.env
.git/
configs/
artifacts/runs/
artifacts/dabench-lc/
data/
old/
Comparison/
.pytest_cache/
**/__pycache__/
**/.ipynb_checkpoints/
```

注意 `artifacts/gateway_caps.yaml` 必须保留进镜像（`tool_calling` 模式启动校验依赖它），所以不能整目录排除。可选方案：

- 在 `.dockerignore` 里用 `!artifacts/gateway_caps.yaml` 反向放行。
- 或把它复制到镜像独立路径（如 `/app/gateway_caps.yaml`），并在提交入口中覆盖 `observability.gateway_caps_path`。

### 6. `langsmith_enabled` 必须在提交时锁死为 false（P0）

`config.py:165` 默认 false，但若意外打开，容器会尝试连 LangSmith → 规则 §5.5 网络封锁导致连接挂起 → 浪费 12h 总预算。

**修复方向**：提交入口强制覆盖 `observability.langsmith_enabled=False`，并在入口启动时 `os.environ.pop("LANGSMITH_API_KEY", None)` 等清理 `LANGSMITH_*` 系列环境变量。

## 三、需要更具体的描述

### A. 总运行时预算（计划只说"调整"）

具体测算：320 题 × 600s ÷ 4 worker = **13.3h > 12h 上限**。给出具体推荐：

- A 榜（约 60 题）：`max_workers=4`、`task_timeout_seconds=600` 没问题。
- B 榜（约 320 题）：建议 `max_workers=8`、`task_timeout_seconds=480`，理论上限约 5.3h，留时间给冷启动 + 失败重试。
- 设全局兜底：到第 11h 触发"放弃剩余任务、等所有 in-flight 写盘"逻辑。规则 §4.2 的 SIGTERM→30s→SIGKILL 不够把 in-flight 子进程清理完。

### B. `MODEL_*` 注入路径（计划只说"动态生成"）

更明确的实现：在 `submission` 入口里直接构造 `AppConfig`，不调用 `load_app_config`（避免任何 yaml 路径），用：

```python
agent = AgentConfig(
    model=os.environ["MODEL_NAME"],
    api_base=os.environ["MODEL_API_URL"],
    api_key=os.environ.get("MODEL_API_KEY", "EMPTY"),
    action_mode="tool_calling",
)
```

并对 `MODEL_API_URL` 缺失抛 `ConfigError`（计划已提）；同时 `MODEL_API_KEY` 缺失允许（vLLM 测试常用 `EMPTY`）—— 计划未区分。

### C. `gateway_caps.yaml` 在镜像内的存放位置

当前 `observability.gateway_caps_path` 默认 `<PROJECT_ROOT>/artifacts/gateway_caps.yaml`（`config.py:166-168`）。镜像内 PROJECT_ROOT 需保持代码布局一致，或在提交入口里把路径改成 `/app/gateway_caps.yaml` 之类的固定位置。计划没指明。

## 四、其他小修

- **`prediction.csv` 编码**：`run/runner.py:148` 用 `encoding="utf-8"`（无 BOM），符合规则。
- **空值规范化**：规则 §6.5 把 `null` / `nan` / `<na>` 等都视为空字符串，当前 `_write_csv` 直接 `csv.writer.writerows`，若 finalize 阶段把 `None` 写成字符串 `"None"`，评测会当真值。建议加测试：`answer.rows` 里的 `None` 必须落盘为空 cell（`csv.writer` 默认行为是空字符串，应该 OK，但需要确认 finalize 不会先 `str(None)`）。

## 六、对计划「推荐结论」的判断

新增最小提交适配层 `data_agent_langchain.submission`、不大改 runner —— **方向正确**，但需要明确：

1. 该模块不能只做扁平化复制 —— 还要兜住 per-task 异常、强写空 prediction.csv、过滤 `/logs` 敏感内容。
2. 不应在 submission 层重新加载 yaml 配置；直接从 env 构造 `AppConfig`。
3. 该模块需要自己的离线 smoke 测试：用 fake LLM + 2-3 个本地 task，断言 `/output/task_X/prediction.csv` 存在且 UTF-8 可解析。

## 七、合规快检（已在代码层验证）

- 工具层无 `requests.` / `httpx.` / `urllib` / `http.client` / `websocket` 的外网调用（`grep` 验证为空）。
- 唯一外部连接是 `ChatOpenAI` → `MODEL_API_URL`，符合规则 §5.5。
- `python_exec` 子进程沙箱 + 30s 硬超时（`tools/python_exec.py:27`、`tools/python_exec.py:155-165`）。
- `filesystem.resolve_context_path` 强制 sandbox 边界，拒绝 `..` 逃逸（`tools/filesystem.py:21-29`）。

## 八、总结

计划覆盖了**容器化与 I/O 路径**两大主线，但漏掉了**异常隔离、`/logs` 隐私脱敏、`.dockerignore` 配方、LangSmith 锁死、`task.json` schema 容错**这五个 P0 级合规点。建议把"二、关键遗漏"的 6 点直接合并进计划 §二与 §六，再细化 §三的具体参数值后再开始 Dockerfile 编写。
