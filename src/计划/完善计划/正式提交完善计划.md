# 正式提交完善计划

本文档梳理 `data_agent_langchain` 按 `rules.zh.md` 进行 KDD Cup 2026 Data Agents 正式 Docker 提交前需要补齐、验证和规避风险的事项。

## 一、当前检查结论

当前 `data_agent_langchain` 已具备任务遍历、单任务运行、子进程超时、`prediction.csv` 写出、Tool Calling 能力校验等基础能力，但距离正式比赛提交还缺少提交专用容器层与官方 I/O 布局适配。

## 二、必须优先完善的问题

### 1. 缺少 Docker 提交入口

当前仓库未发现可用于正式提交的 `Dockerfile`、`.dockerignore` 或容器入口脚本。比赛规则要求提交的是可直接运行的 Docker 镜像，评测系统会通过 `docker run` 注入 `/input`、`/output`、`/logs` 与 `MODEL_*` 环境变量。

需要补齐：

- `Dockerfile`：构建 linux/amd64 可运行镜像。
- `.dockerignore`：排除缓存、artifacts、本地密钥配置和大文件。
- 容器入口：无需额外参数即可启动完整 benchmark。

### 2. `/output` 目录布局不符合官方最终要求

当前 runner 默认输出结构为：

```text
run.output_dir/<run_id>/<task_id>/prediction.csv
```

比赛要求输出结构为：

```text
/output/<task_id>/prediction.csv
```

不能多一层 `run_id`。建议新增提交专用入口，在内部保留 run 目录用于 trace / metrics，然后把每个任务的 `prediction.csv` 扁平复制到 `/output/<task_id>/prediction.csv`。

### 3. 模型配置需要严格从官方环境变量读取

规则要求正式评测时只能通过运行时环境变量读取模型配置：

- `MODEL_API_URL`
- `MODEL_API_KEY`
- `MODEL_NAME`

当前 `build_chat_model()` 使用 `config.agent.api_base`、`config.agent.api_key`、`config.agent.model`。这可以保留，但提交入口必须从 `MODEL_*` 动态生成安全配置，不能依赖本地 YAML 中的真实密钥。

### 4. `gateway_caps.yaml` 需要提交态策略

当前 `agent.action_mode="tool_calling"` 时会强校验 `gateway_caps.yaml` 存在并且 `tool_calling=true`。如果正式镜像中缺少该文件，容器会在运行前失败。

建议策略：

- 在提交镜像内提供不含敏感信息的 `gateway_caps.yaml`。
- 内容按官方规则中“支持 OpenAI 风格 Tool Calling”的能力设置。
- 同时保留 `json_action` fallback 作为本地调试选项，但正式提交优先使用 `tool_calling`。

### 5. 日志需要写入 `/logs`

规则建议将 stdout/stderr 同步写入 `/logs/runtime.log`，否则容器退出后调试信息可能丢失。

需要补齐：

- 容器入口统一写 `/logs/runtime.log`。
- 将 summary、trace 或必要错误信息保存到 `/logs` 或内部运行目录。
- 避免日志中泄漏 API key、测试集标准答案或其他敏感信息。

## 三、建议同步完善的问题

### 1. 依赖固化

当前 `pyproject.toml` 可见依赖中没有显式列出 LangChain / LangGraph 相关包，但源码依赖：

- `langchain-core`
- `langchain-openai`
- `langgraph`

正式 Docker 构建时应确保这些依赖被显式安装并版本兼容，避免本地 conda 环境可运行但镜像内缺包。

### 2. CLI 文案同步

`gateway-smoke` 命令末尾仍有“默认 action_mode remains json_action until Phase 6 default-switch approval”的过期输出；当前 README / config 已显示默认切到 `tool_calling`。该问题不阻断提交，但建议同步，避免误导。

### 3. 总运行时预算

规则要求整个容器运行总时长不超过 12 小时。当前单任务默认超时为 600 秒，`max_workers=4`。正式提交前应结合任务规模调整：

- `run.max_workers`
- `run.task_timeout_seconds`
- `agent.max_steps`
- `agent.max_replans`

目标是在单任务失败时不中断全局运行，并尽快写出已完成任务结果。

## 四、推荐实施顺序

### 第 1 步：新增提交专用入口

建议新增 `data_agent_langchain.submission` 模块，职责为：

1. 校验 `/input`、`/output`、`/logs` 可用。
2. 从 `MODEL_API_URL`、`MODEL_API_KEY`、`MODEL_NAME` 生成运行配置。
3. 调用现有 `run_benchmark()`。
4. 将内部 run 目录下的 `prediction.csv` 扁平复制到 `/output/<task_id>/prediction.csv`。
5. 将运行日志和 summary 写入 `/logs`。

### 第 2 步：新增测试

建议先加测试再实现，至少覆盖：

- 缺少 `MODEL_API_URL` 时入口给出明确错误。
- 入口生成的配置使用 `/input` 作为 dataset root。
- 内部结果可正确扁平化到 `/output/<task_id>/prediction.csv`。
- 不会把输出写成 `/output/<run_id>/<task_id>/prediction.csv`。

### 第 3 步：新增 Docker 文件

建议新增：

- `Dockerfile`
- `.dockerignore`
- 提交用无密钥配置模板或入口动态生成配置
- 默认 `ENTRYPOINT`，确保 `docker run` 无需额外参数即可执行。

### 第 4 步：容器级 smoke 验证

正式打包前至少验证：

```text
docker run --rm \
  -v <local_input>:/input:ro \
  -v <local_output>:/output:rw \
  -v <local_logs>:/logs:rw \
  -e MODEL_API_URL=<模型服务地址> \
  -e MODEL_API_KEY=<密钥> \
  -e MODEL_NAME=qwen3.5-35b-a3b \
  <team_id>:v<N>
```

验证重点：

- `/output` 下只有一层 `task_<id>` 目录。
- 每个成功任务都有标准 UTF-8 `prediction.csv`。
- `/logs/runtime.log` 存在。
- 容器无交互输入要求。

## 五、正式提交检查清单

- 镜像名符合 `<team_id>:v<N>`。
- 归档名符合 `<team_id>_v<N>.tar.gz`。
- 使用 `docker save` 导出，不使用 `docker export`。
- 镜像支持 `linux/amd64`。
- 归档大小小于 10 GB。
- 镜像内不包含本地 API key 或敏感 YAML。
- 代码不硬编码 `MODEL_API_URL`、`MODEL_API_KEY`、`MODEL_NAME` 的真实值。
- 容器只访问 `MODEL_API_URL` 指定的内部模型服务。
- 容器不修改 `/input`。
- 输出路径为 `/output/<task_id>/prediction.csv`。
- 日志写入 `/logs`，且不泄漏标准答案或敏感信息。
- Google Drive 链接权限为 Anyone with the link / Viewer。
- 提交邮件从队长注册邮箱发送。

## 六、建议优先级

| 优先级 | 事项 | 原因 |
| --- | --- | --- |
| P0 | Dockerfile / ENTRYPOINT | 没有它无法正式提交 |
| P0 | `/output` 扁平化 | 路径不合规会直接影响评分 |
| P0 | `MODEL_*` 环境变量接入 | 规则硬性要求，避免硬编码密钥 |
| P0 | `/logs/runtime.log` | 失败后排查依赖日志 |
| P1 | gateway caps 提交态文件 | `tool_calling` 默认模式依赖它 |
| P1 | 依赖固化 | 防止镜像内缺包 |
| P2 | CLI 文案同步 | 降低维护误导 |

## 七、推荐结论

建议采用“提交专用入口层”方案：不大改现有 runner，新增最小容器适配层来满足比赛规则。这样既能保留当前 LangGraph 后端的开发输出结构，又能保证正式评测看到的是 `/output/<task_id>/prediction.csv` 的标准布局。
