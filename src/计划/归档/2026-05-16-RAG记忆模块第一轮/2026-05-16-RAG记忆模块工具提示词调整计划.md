# RAG/记忆模块建立后的工具与提示词调整计划

## 1. 背景与目标

当前 RAG 与记忆模块已经接入 agent 流程：

- `planner_node.py` / `task_entry_node.py` 负责召回 dataset facts 与 corpus snippets。
- `model_node.py` 将 `memory_hits` 渲染后追加到 LLM messages。
- `tool_node.py` 在数据预览类工具成功后写入 dataset knowledge。
- 工具层仍沿用原有 legacy 工具，不直接感知 RAG/记忆召回结果。

本计划目标是分析并调整 RAG/记忆建立后对工具调用与提示词的影响，降低 stale memory、错误路径、错误 schema 和重复工具调用造成的失败。

## 2. 当前接入方式结论

### 2.1 Dataset memory

写入路径：

- `tool_node.py` 在 `read_csv`、`read_json`、`read_doc`、`inspect_sqlite_schema` 成功后写入 dataset knowledge。

召回路径：

- `planner_node.py` 和 `task_entry_node.py` 调用 `recall_dataset_facts()`。
- namespace 当前使用 `dataset:{Path(dataset_root).name}`。

风险：

- benchmark 下多个任务的 `dataset_root` 末级目录相同，常见为 `input`。
- 因此不同任务共享 `dataset:input`，容易跨任务污染。

### 2.2 Corpus RAG

召回路径：

- `recall_corpus_snippets()` 使用 `corpus_task:{task_id}`。
- 每个 task 使用当前 task corpus handles 召回。

风险相对较低：

- namespace 是 task-scoped。
- 主要风险来自 prompt 中未明确说明 snippets 的权威等级。

### 2.3 Prompt 注入

当前 prompt 注入方式：

- Dataset facts 渲染为 `## Dataset facts (from prior runs, informational only)`。
- Corpus snippets 渲染为 `## Reference snippets (from task documentation)`。
- 两者作为额外 `HumanMessage` 追加到 messages 末尾。

问题：

- 当前提示词没有明确说明这些内容与 `list_context`、工具 observation 的权威顺序。
- 模型可能把 prior-run dataset facts 当成当前任务真实文件清单。

### 2.4 工具层

当前工具层特点：

- 工具不感知 recall hits。
- `list_context` 结果没有形成硬约束。
- `known_paths` 字段存在于 `RunState`，但当前未充分维护和使用。
- 缺失路径、错误表名等失败容易被模型原样重复。

## 3. 失败证据与根因判断

### 3.1 task_355

现象：

- `metrics.json` 显示 dataset memory 召回 namespace 为 `dataset:input`。
- 命中了 stale path，例如 `dk:input:csv/member.csv`。
- 当前 task 的实际 context 中不存在 `csv/member.csv`。
- agent 后续大量重复 `read_csv("csv/member.csv")`，最终失败。

判断：

- 主要根因是 dataset memory namespace 过宽。
- 次级原因是 prompt 未强调 recalled path 必须由当前 `list_context` 验证。
- 第三级原因是工具层缺少重复缺失路径防循环机制。

### 3.2 task_350

现象：

- dataset memory 命中多个跨任务文件事实。
- `read_csv` 调用次数异常高。

判断：

- 与 task_355 类似，属于 stale dataset facts 诱导路径选择。

### 3.3 task_408

现象：

- DB/SQL 类调用重复，`execute_context_sql` 调用次数高。
- 模型可能反复尝试错误表名或错误 schema 假设。

判断：

- 主要是 schema observation 没有被强约束使用。
- SQL 工具失败反馈需要提供可操作纠偏信息。

### 3.4 task_418

现象：

- dataset memory 命中 `dk:input:db/results.db`。
- trace 中反复出现 `Missing context asset: db/results.db`。
- `inspect_sqlite_schema` 调用次数异常高。

判断：

- 再次证明 `dataset:input` 污染会直接诱导错误路径循环。

## 4. 总体判断

需要调整，且优先级不是先调 RAG 检索参数，而是：

1. 修正或隔离 dataset memory namespace。
2. 明确 prompt 中 RAG/记忆内容的权威等级。
3. 让工具层把当前 context 和 schema observation 变成硬约束。
4. 增加重复失败的 loop guard。
5. 最后再评估 RAG top-k、budget、多次召回等检索参数。

### 4.1 计划审查补强点

本计划原始版本的方向正确，但落地前需要补强以下内容：

- **召回时序**：当前 Plan-and-Solve 的 plan 生成先于 recall，不能假设 planner 已经看到了 corpus snippets 或 dataset facts。
- **污染迁移**：即使改了 namespace，旧的 `dataset:input` 记录仍在 store 中，需要明确“不自动注入旧 namespace”与回滚策略。
- **路径硬约束边界**：`known_paths` 应只来自当前 task 的 `list_context`，并且需要明确尚未 discovery 时的行为。
- **指标可量化**：验收标准应从“明显下降”改为可计算的 retry count、tool call count、token count 和成功率变化。
- **实施粒度**：需要拆成可独立验证的小任务，避免一次性改 memory、prompt、tools、advance_node 导致无法归因。
- **与 R6 边界**：本计划处理 memory/RAG 使用契约与 stale-memory guard；R6 继续处理 corpus recall precision，不应混成一个大改造。

### 4.2 关键架构细节：Plan 阶段当前看不到召回结果

当前 `planner_node.py` 的流程是先生成 plan，再执行 dataset/corpus recall 并把 hits 写入 `memory_hits`。因此：

- Plan-and-Solve 的初始 plan 不能依赖 RAG/memory 已注入。
- `render_dataset_facts()` / `render_corpus_snippets()` 的提示词调整主要影响后续 execution/model step。
- 如果未来要让 planner 使用 corpus snippets，应单独设计“planning-before-recall”改造，并只允许 task-scoped corpus snippets 进入 planner。
- 在 dataset memory 污染未解决前，不应让 prior-run dataset facts 进入 planner。

## 5. 设计原则

### 5.1 权威顺序

建议固定如下信息权威等级：

1. 当前 task 的实时工具 observation。
2. 当前 task 的 `list_context` 文件树。
3. 当前 task 的 schema inspection / read preview。
4. 当前 task corpus snippets。
5. prior-run dataset facts。

如果低权威信息与高权威信息冲突，必须信任高权威信息。

### 5.2 路径使用规则

- 任何文件路径必须先出现在当前 task 的 `list_context` 输出中。
- memory 或 RAG 中提到的路径只能作为候选，不能直接作为工具调用依据。
- 如果工具返回 missing context asset，不允许原样重试同一路径。

### 5.3 Schema 使用规则

- SQL 必须以当前 task 的 `inspect_sqlite_schema` 为依据。
- 文档、memory、RAG 中提到的表名或列名必须由 schema inspection 验证后才能使用。
- `no such table` / `no such column` 后不能原样重试 SQL。

### 5.4 不破坏 gate 规则

现有 gate 规则要求：

- `execute_python`、`execute_context_sql`、`answer` 前必须先完成数据预览。
- `list_context` 和数据预览动作是 agent 进入计算/回答前的安全前置条件。

本计划中的路径约束、schema 约束和 loop guard 是 gate 的增强，不应绕开或弱化 gate。

## 6. 分阶段调整计划

### Phase 0：冻结问题定义与评估样本

目标：建立稳定验证集，避免只针对单个 trace 过拟合。

建议样本：

- `task_350`：高频 `read_csv`，疑似 stale dataset facts 路径诱导。
- `task_355`：典型 `csv/member.csv` stale path 循环。
- `task_408`：SQL/schema 循环。
- `task_418`：典型 `db/results.db` stale path 循环。

验收指标：

- missing path 同参重复调用次数：每个 task、每个 path 不超过 1 次原样重试。
- no such table / no such column 同参重复调用次数：每个 task、每个 SQL signature 不超过 1 次原样重试。
- 总 tool call 次数：在失败样本上下降，且不以跳过必要 preview 为代价。
- 不增加 gate bypass。
- 不降低原有成功任务的通过率。

基线统计建议：

- `missing_context_asset_retry_count`：同一 path 的 missing context asset 次数。
- `schema_mismatch_retry_count`：同一 SQL signature 或同一不存在表/列的重复次数。
- `stale_memory_path_count`：memory hit 中出现、但当前 `list_context` 未出现的 path 数。
- `tool_call_count_by_tool`：按工具统计调用次数。
- `prompt_tokens` / `completion_tokens` / `wall_clock_s`。

### Phase 1：处理 dataset memory namespace 污染

目标：阻断 `dataset:input` 级别跨任务污染。

候选方案：

#### 方案 A：暂时不把 dataset facts 注入 prompt

做法：

- 保留写入 dataset memory。
- 保留 corpus RAG 注入。
- 暂时不把 dataset facts 作为 `HumanMessage` 注入模型。

优点：

- 风险最低。
- 能快速验证失败循环是否主要由 dataset facts 污染引起。

缺点：

- 暂时失去 dataset memory 对跨 run 的帮助。

推荐程度：高，适合作为第一步 A/B。

#### 方案 B：改为 task-scoped dataset memory

做法：

- dataset knowledge namespace 改为 `dataset_task:{task_id}` 或等价 task-scoped 形式。
- 写入和召回都使用 task-level namespace。

优点：

- 避免跨任务污染。
- 保留同一 task 重跑时复用价值。

缺点：

- 跨 task 泛化收益下降。
- 需要同步调整写入、召回、metrics 预期。

推荐程度：高，适合作为正式修复方向。

#### 方案 C：保留跨任务 dataset memory，但召回后按当前 context 过滤

做法：

- `list_context` 后才允许注入 dataset facts。
- 只保留路径出现在当前 context 的 facts。

优点：

- 仍可能利用跨任务经验。

缺点：

- 实现复杂。
- Plan 阶段在 `list_context` 前无法安全使用。
- 若 facts 不含标准路径字段，过滤会不稳定。

推荐程度：中，适合后续 shared memory 设计，不建议作为第一步。

Phase 1 推荐落地顺序：

1. 先做方案 A 的开关或实验配置。
2. 对失败样本重跑，确认 stale dataset facts 的影响。
3. 再做方案 B 作为长期修复。

迁移与回滚要求：

- 不直接删除旧 store 中的 `dataset:input` 记录，先通过读取侧隔离降低风险。
- metrics 中应能区分 dataset memory recall 与 corpus recall，便于确认变更是否生效。
- 若 Variant A 禁用 dataset facts 后成功率下降，应可通过配置回滚到原行为做对照。
- 长期若采用 task-scoped namespace，旧 namespace 只用于离线分析或后续 curated shared memory，不再默认 prompt 注入。

### Phase 2：完善 RAG/记忆提示词契约

目标：让模型明确知道 memory/RAG 是辅助信息，不是当前任务事实源。

建议调整位置：

- `render_dataset_facts()`。
- `render_corpus_snippets()`。
- 必要时补充 `REACT_SYSTEM_PROMPT` / `PLANNING_INSTRUCTION`。

提示词应表达：

- Dataset facts are from prior runs and may be stale.
- Reference snippets are documentation references, not verified file inventory.
- Current `list_context` and tool observations override memory/RAG.
- Do not call tools on a path mentioned only by memory/RAG.
- If a path is missing once, do not retry it unchanged.
- If docs mention data but no CSV/DB exists, read `doc/*.md` or `knowledge.md` as data sources.

注意事项：

- 不优先修改 `tools/descriptions.py`，因为该文件有 legacy parity 风险。
- 优先在 memory rendering 或 system-level memory policy 中补充规则。
- 提示词规则应使用英文表达，保持与现有 tool/system prompt 风格一致。
- Dataset facts 和 corpus snippets 应分别说明来源、时效性、是否可作为路径依据。
- 如需要共用规则，优先抽成单一 memory authority policy 文本，避免多处 prompt 漂移。

### Phase 3：让 `list_context` 结果成为路径硬约束

目标：工具层阻断 stale memory path。

建议做法：

- `list_context` 成功后，把当前 context 的文件路径写入 `RunState.known_paths`。
- `read_csv`、`read_json`、`read_doc`、`inspect_sqlite_schema`、`execute_context_sql` 在执行前检查目标路径是否属于 known paths。
- 如果 path 不在 known paths：
  - 返回 validation-style error。
  - 附带可用路径候选。
  - 明确提示不要原样重试。

示例反馈语义：

```text
Path csv/member.csv is not present in the current task context.
Available files include: csv/expense.csv, doc/member.md, knowledge.md.
Do not retry csv/member.csv unless list_context later shows it exists.
```

注意：

- 该机制不应替代 gate，而是强化 gate。
- 如果尚未 `list_context`，继续保留原有 “Call list_context first” 语义。
- 如果已经 `list_context`，错误消息应改为“当前 context 不存在该路径”，而不是继续要求 list_context。
- `known_paths` 应只包含当前 task context 内的文件路径，不应包含目录路径或 memory/RAG 中未验证路径。
- 路径应统一为相对 POSIX 风格，例如 `csv/member.csv`，避免 Windows 反斜杠导致同一文件多种签名。
- `execute_context_sql` 的数据库文件参数应受 `known_paths` 约束；SQL 文本本身不应被路径约束误伤。
- 如果 `known_paths` 为空且 `discovery_done` 为 false，应触发 discovery/gate 提醒，而不是把所有路径都判为 stale。

### Phase 4：增加重复失败 loop guard

目标：避免同一错误调用消耗几十个 step。

建议规则：

- 同一 `tool + normalized_input` 出现 missing path 一次后，下一次相同调用直接短路为更强提醒。
- 同一 `tool + normalized_input` 连续失败两次后，触发 replan 或 forced alternative action。
- SQL 的相同 query 连续失败后，强制 inspect schema 或换 query。

可记录字段：

- `failed_action_signatures`。
- `last_failed_path`。
- `last_failed_sql_signature`。
- 或从 `steps` 中动态推导，避免新增 state 字段。

建议优先从 `steps` 动态推导，降低状态迁移成本。

边界：

- loop guard 只针对确定性失败，如 missing path、no such table、no such column、tool validation。
- 不应拦截 `answer`，也不应拦截一次性成功后的正常后续分析。
- 不应把 transient timeout 与 deterministic validation error 混为一类。

### Phase 5：SQL/schema 纠偏增强

目标：降低 `task_408` 类 schema 循环。

建议做法：

- `execute_context_sql` 遇到 `no such table` / `no such column` 时：
  - 返回错误类型为 validation/schema mismatch。
  - 附带已知表名或建议先 `inspect_sqlite_schema`。
- `inspect_sqlite_schema` 成功后，将 schema observation 作为 pinned preview step 保留在 scratchpad。
- 如果 SQL 引用未在 schema 中出现的表名，考虑提前阻断。

注意：

- 不能禁止复杂 SQL。
- 只针对明显不存在的表/列做纠偏。
- 若 schema inspection 已成功，应在错误提示中引用可用表名；若尚未 inspection，应强制建议先 inspection。

### Phase 6：重新评估 RAG 检索策略

目标：在污染和循环问题缓解后，再判断 RAG 本身是否需要调参。

评估内容：

- `retrieval_k` 是否过高或过低。
- `prompt_budget_chars` 是否合理。
- 是否需要 planner 前召回。
- 是否需要失败后动态再召回。

建议：

- 不在 Phase 1 前扩大 `retrieval_k` 或 budget。
- 不在 dataset memory 污染解决前做多次自动 recall。
- R6 corpus recall evaluation 继续推进，但需新增 stale path / memory conflict 相关指标。

## 7. 与现有 R6 / M4 follow-up 计划关系

### 7.1 与 R6 corpus recall evaluation 的关系

本计划不替代 R6。

R6 主要评估 corpus recall 的 precision、hit quality 和 A/B 策略。本计划补充的是：

- RAG/memory 进入 prompt 后如何被 agent 使用。
- recall 命中与当前工具 observation 冲突时如何处理。
- stale dataset facts 是否诱导错误路径。

建议给 R6 增加指标：

- `memory_conflict_count`：召回路径不在当前 context 中的次数。
- `stale_path_retry_count`：missing path 后同一路径重试次数。
- `schema_mismatch_retry_count`：no such table/column 后同 SQL 或同表名重试次数。
- `rag_used_after_verification_count`：RAG 提示经工具验证后被正确使用的次数。

### 7.2 与 corpus-rag-m4-followups 的关系

M4 follow-up 中已有 shared corpus、timeout、Dockerfile、R6 等任务。

本计划建议优先插入一个新主题：

- `RAG/memory authority contract and stale-memory guard`

定位：

- 在 shared corpus 扩大召回范围前，先定义 memory/RAG 的权威边界。
- 在多次 recall 或 shared memory 前，先避免已有 dataset memory 污染继续放大。

## 8. 不建议立即做的事

- 不建议先扩大 `retrieval_k` 或 `prompt_budget_chars`。
- 不建议马上新增 `search_memory` 工具。
- 不建议只改 prompt 而不改工具反馈。
- 不建议把 dataset facts 与 corpus snippets 混成同一权威级别。
- 不建议优先修改 `tools/descriptions.py`，除非 parity 测试明确允许。
- 不建议在未解决 stale dataset memory 前做失败后自动多次 recall。

## 9. 推荐实施顺序

推荐顺序：

1. 建立失败样本基线：`task_350`、`task_355`、`task_408`、`task_418`。
2. 暂停或隔离 dataset facts prompt 注入。
3. 给 memory/RAG prompt 增加权威等级规则。
4. 把 `list_context` 结果维护为 `known_paths`。
5. 对 missing path 增强工具反馈。
6. 对 no such table / no such column 增强 SQL 反馈。
7. 增加重复失败 loop guard。
8. 重跑失败样本和一组原成功样本。
9. 再进入 R6 recall quality / retrieval 参数评估。

## 10. 验证计划

### 10.1 单元测试建议

建议新增或扩展测试覆盖：

- dataset memory namespace 不再使用裸 `dataset:input` 作为跨任务自动注入来源。
- `render_dataset_facts()` 包含 stale/verification policy。
- `render_corpus_snippets()` 包含 reference authority policy。
- `list_context` 成功后可提取 current context paths。
- missing path 错误消息包含 available files 和 do-not-retry 提示。
- SQL schema mismatch 错误包含 inspect schema 或 available tables 提示。

### 10.2 回归任务验证

至少重跑：

- `task_350`
- `task_355`
- `task_408`
- `task_418`

观察指标：

- `read_csv` / `inspect_sqlite_schema` / `execute_context_sql` 高频重复是否下降。
- token 使用量是否下降。
- replan 是否减少或更有效。
- 是否出现 gate 规则绕过。
- 是否出现原成功任务变失败。

### 10.3 对照实验建议

建议做三组对照：

- Baseline：当前版本。
- Variant A：禁用 dataset facts 注入，仅保留 corpus snippets。
- Variant B：task-scoped dataset memory + prompt authority policy + missing path guard。

优先比较：

- 成功率。
- 平均 tool calls。
- missing path retry count。
- schema mismatch retry count。
- prompt tokens。

### 10.4 验证命令与证据要求

每次改造完成后，需要保留以下证据：

- 变更前后 `metrics.json` 中 `tool_calls`、`memory_recalls`、`tokens`、`wall_clock_s` 的对比。
- 变更前后 `trace.json` 中 missing path / schema mismatch 的重复次数对比。
- 至少一个原成功任务的回归结果，确认没有因 hard block 误伤正常路径。
- 如果只做 prompt 变更，需要同时记录是否仍出现同一路径/SQL 原样重试，避免把改善误判为工具层修复。

命令层面建议使用现有 benchmark/对比脚本，不在本计划中新增一次性私有流程：

```powershell
dabench-lc run-benchmark --config configs/local.yaml --graph-mode plan_solve --memory-mode read_only_dataset --memory-rag
python bench_comparison.py
```

## 15. 风险与回滚策略

### 15.1 主要风险

- **过度 hard block**：某些任务可能在未完成 `list_context` 的情况下需要工具纠错，过早阻断会降低探索能力。
- **prompt drift**：如果规则分散在多个 prompt 片段中，后续维护容易不一致。
- **memory 收益下降**：禁用 dataset facts 可能让部分同 task 重跑收益下降。
- **metrics 解释困难**：同时改 prompt、memory namespace、tool guard 会让效果归因不清。

### 15.2 回滚策略

- 所有第一轮行为应尽量通过配置或小范围函数开关控制。
- 若 hard block 误伤，应回退为 warning-style validation，再保留 do-not-retry 提示。
- 若禁用 dataset facts 后成功率明显下降，应切换为 task-scoped dataset facts，而不是恢复裸 `dataset:input` 自动注入。
- 若 prompt 改动导致 parse/tool-call 格式退化，应优先回滚 system prompt 改动，只保留 memory rendering 中的短 policy。
